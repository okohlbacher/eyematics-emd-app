### K1 — Trajectory tooltips: two tooltips now; rework around the active layer (I1/J1 follow-up)
Tester: "Now there are TWO tooltips — one for the nearest on the x-axis and one for the scatter point I hover over… Change to: show scatter tooltip only when scatter points are activated; when individual lines are activated, highlight the lines on hover; when scatter tooltips are not activated → tooltip when hovering over a line; **remove the x-axis related tooltips**." + "the curve settings button is not self-explanatory."
**Diagnosis:** v1.15 added the imperative scatter tooltip but LEFT the Recharts axis `<Tooltip>` for non-scatter series → two tooltips coexist. NF wants a single, layer-aware hover tooltip and NO axis tooltip.
**Proposed measures:**
- **K1a:** remove the axis `<Tooltip>` entirely.
- **K1b:** scatter hover tooltip only when the Streupunkte layer is on (already imperative — gate it on `layers.scatter`).
- **K1c:** when the per-patient-lines layer is on, hovering a line **highlights that line** (emphasis) AND shows a tooltip for that patient — but only when scatter is OFF (scatter takes precedence when both on).
- **K1d:** make the "curve settings" (Einstellungen/Anzeigeebenen) button self-explanatory — clearer label/icon/tooltip.
**Confidence:** medium; imperative hover infra exists (hoveredDatumRef + imperative tooltip), extend to lines + gate by layer.

### K2 — Performance STILL not acceptable on settings change (I2/J2 follow-up, HIGH, recurring 4th round)
Tester: "Performance did not improve significantly. Dashboard still unresponsive while loading plots **when changing settings**." Backlog response: "Current behaviour is not responsive."
**Diagnosis:** v1.15 moved AGGREGATION to a worker + progressive first mount, but **toggling a layer / changing settings re-renders all panels synchronously on the main thread** — and the dominant remaining cost is **Recharts building SVG for many nodes** (scatter ≤1500×2 circles ×3 panels, and/or per-patient lines), which a worker cannot help. Progressive mount helped the FIRST load but a settings change still triggers a synchronous re-render burst.
**Proposed measures (need a decision):**
- **Profiling spike first** to pin the exact cost on a settings change (scatter SVG vs per-patient `<Line>` count vs compare-mode multi-cohort).
- Likely substantive fix: **canvas rendering** for the heavy scatter (and possibly per-patient lines) instead of thousands of SVG nodes — the SVG node count is the wall Recharts keeps hitting. OR debounce settings changes + re-stage the progressive mount on every settings change so the toggle returns instantly and panels rebuild progressively.
**Confidence:** high it's still broken; the worker/progressive path has not satisfied the tester for 3 rounds → the rendering layer (SVG node count) is the real bottleneck. **Decision needed on the heavier approach (canvas).**
