## 🔴 Bugs (code-verified root causes)

### A1 — FALL-010 drill-down click still doesn't navigate (HIGH)
Tester: points visible, tooltip shows pseudonym, **click does nothing**.
**Diagnosis:** the tooltip works via Recharts' chart-level proximity events, but the symbol-level `<Scatter onClick>` evidently never fires in the real browser (median `<Line>`s render after/over the scatter; symbol hit-targets tiny; Recharts v3 symbol event binding unreliable). Our RTL test mocks recharts, so it can't catch this.
**Fix:** move drill-down to the **chart-level `onClick` on `ComposedChart` using `activePayload`** — the same event pipeline that demonstrably works (it powers the tooltip the tester sees). Click anywhere near a point → navigate to `activePayload`'s patient. Keep IDOR gate. Verify in a live browser this time, not only RTL.

### A2 — DocQuality counts/metrics don't react to the time filter (HIGH; pairs with B1)
Tester: "values are not adapted when I change the time range… the metric is not providing any information."
**Code-confirmed:** `DocQualityPage.distinctPatientCount` = `new Set(cases…)` over the **unfiltered** case set; and `filterCasesByTimeRange` only trims observations — it never drops cases, so per-center `patientCount`/denominators stay constant.
**Fix:** scope **everything** on the page to the selected window per B1 (see below): case-level inclusion (≥1 obs in window) for Grundgesamtheit + per-center counts; metric computation over windowed observations only.

### A3 — FALL-011 overlay distorts the X-axis; applies to one plot only (MEDIUM)
Tester: enabling the overlay changes the X-range, chart "hardly readable"; expected a relative-since-start axis.
**Diagnosis:** the month-binned reference points inject additional date categories into the category X-axis, stretching the domain. Needs live verification.
**Fix:** clamp reference points strictly to the patient's existing date domain (emit reference values only for the patient's own x-categories — the bin lookup already supports this), and/or render the case trajectory on a relative-time axis as the tester expects. Decide single- vs multi-plot scope with C-scope (tester noted it "only changes one plot" — by design it's VisusCrtChart only; confirm that's acceptable).

### A4 — FALL-012 label leftovers (SMALL)
Tester: (a) "Visus vs CRT" plot (DistributionCharts scatter) still unlabeled/unclear; (b) Y-axis label "Visus (Dezimal, bestkorrigiert)" is cut off — move to a legend under the X-axis where there's space; (c) **no open circle is ever visible**, so the interpolation note refers to nothing.
**Fix:** label the Visus-vs-CRT scatter; relocate the long Visus label to a legend line; investigate whether interpolated points are actually rendered as open circles — if interpolation markers don't (or rarely) render, show the hint **conditionally** (only when interpolated points exist) or fix the marker rendering.

### A5 — Audit log: 401 burst shown as `unauthenticated` (MEDIUM, security-adjacent)
Tester screenshot: ~10 identical `401 · Sonstige Aktion · unauthenticated` rows at the same second (11:00:46), then `admin` succeeds.
**Diagnosis:** an **expired-session burst** — the app fires its parallel data loads with an expired access token; all 401; the silent refresh then recovers. Working as designed, but noisy and uninformative.
**Fix (proposed):** for 401s carrying a JWT that is **signature-valid but expired**, attribute the audit actor to the token's username with a marker, e.g. `admin (Sitzung abgelaufen)` — never trust an invalid-signature token (those stay `unauthenticated`). Optionally also collapse same-second duplicate 401s per path. This makes the log readable AND keeps forged-token attempts visible.

### A6 — "Verläufe" tab is very slow (MEDIUM, investigate first)
Tester general remark. Candidates: hundreds of per-patient SVG `<Line>`s, scatter point count, `cohortReference`/cross-cohort memo recomputation, no memoized downsampling. **Action:** profile in browser first, then optimize (e.g. cap/downsample per-patient lines, memoize series, virtualize). Don't guess-optimize.

## 🔵 Decided by tester (former open questions → requirements)

### B1 — Q2 answered: time filter must scope the doc-quality metrics themselves
"If I want to check whether my quality increased/decreased for the last 6 months, I should only see metrics based on values from that time range." → Reverses our "full-history" default: **all Dokumentationsqualität metrics (incl. Vollzähligkeit/completeness, plausibility, per-center scores) computed from windowed data only.** Plus: **more time-range options** than all/6m/1y (e.g. 3m, 12m, custom range).

### B2 — Vollzähligkeit naming collision (from #3 result)
"Vollzähligkeit is computed different to the landing page."
**Code-confirmed:** LandingPage *Datenvollzähligkeit* = consented/total (consent+stub model); DocQuality *completeness/dataCompleteness* = documentation-completeness metrics. Same German word, different semantics.
**Fix:** rename/label distinctly (e.g. "Dokumentations-Vollständigkeit" vs "Einwilligungsquote/Datenvollzähligkeit") + an ℹ definition tooltip on each; align computation behavior with B1.

## 🟣 Design proposals (discuss/design first, then build)

### C1 — Quality case-review workflow rework (#4 follow-up)
Tester: two redundant approve controls is questionable — proposal: **confirm results directly in the "zu prüfende Messwerte" list**; highlight auffällige Werte/fehlende Daten **inside** that list (today only in a separate card); clarify "Behoben" (is it the opposite of confirming? no edit option exists → not self-explanatory); page feels overloaded.
**Suggestion:** a focused design pass on QualityCaseDetail: inline per-measurement confirm/flag actions in the list, status vocabulary (bestätigt/auffällig/behoben) with definitions, remove the duplicated top/bottom control once inline actions exist.

### C2 — Move quality-parameter selection to the Datenqualität tab (Additional point 1)
Tester: it's not self-explanatory why quality-check parameters are configured in Kohortenbildung. **Suggestion:** relocate (or mirror) the qualityParams checklist into the Datenqualität cohort-scope panel — selection still persists with the SavedSearch (D2 unchanged), but it's edited where it's used.

### C3 — Sub-cohort "Split by" wizard (Additional point 2)
Tester: creating sub-cohorts should support the comparison intent directly — **split a cohort by gender / diagnosis / center, or by ranges of age / Visus / CRT**, generating the sub-cohorts in one step.
**Suggestion:** a "Kohorte aufteilen" dialog producing `Parent:Sub` cohorts (existing v1.10 subcohort convention) from a chosen attribute or range list. Nice fit with the existing compare infrastructure.
