## Accumulated Context

### Inherited from v1.0 / v1.1

- Local vs Blaze bundle discrimination: Organization entry presence. Local per-center files each have one Organization; Blaze synthetic bundles do not.
- Server-side FHIR bundle loading with center filtering is authoritative — client never sees unauthorized center data.
- i18n keys live in a single `src/translations.ts` with `{de, en}` pairs (not split de.ts/en.ts).
- `fhirApiPlugin` validates auth via `validateAuth()` from `utils.ts`; centers read from decoded JWT payload.
- Center configuration loads from `data/centers.json` into `server/constants.ts`; fallback `DEFAULT_CENTERS` exists for when the file is missing or malformed.
- Client-side `CENTER_SHORTHANDS` map in `src/services/fhirLoader.ts` duplicates the roster — both must stay in sync (Phase 7 addresses this).

### New for v1.5

- Site roster change is breaking for any persisted cohort/user referencing removed centers — server startup runs a migration (Phase 7 will define it).
- Synthetic data for the 5 new sites is generated by a reproducible Node script, not hand-authored, so re-running with different patient counts / seeds is cheap.
- Cohort outcome analysis is client-side aggregation over already-authorized data; no new cross-site data path is introduced.
