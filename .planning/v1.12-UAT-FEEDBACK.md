## Part 3 — Final UAT round (Phase 45): what I fixed this round

### #3 — QUAL-023 absolute counts on Dokumentationsqualität  🟡 Fixed (code)
**Found:** the **Datenqualität** page already showed absolute counts (Phase 41), but the **Dokumentationsqualität** page (`DocQualityPage`) showed only the percentage prominently — the patient count was a 10px, opacity-60 footnote on `MetricCard`. This is the likely page where you "couldn't find the absolute values."
**Fixed (`fcf756d`):** added a prominent "Grundgesamtheit" population label above the metric grid (total patients in the active time range), and promoted `MetricCard`'s patient count from a footnote to a readable `text-sm font-medium` line beside the %. Both quality pages now surface absolute counts.
**Your check:** confirm the absolute numbers are now obvious on whichever page you were on.

### #8 — A-06 missing axis ticks  🟡 Fixed broadly (code)
**Found:** only `VisusCrtChart` had explicit tick config; 7 other chart components relied on Recharts defaults, which can drop ticks on tight layouts.
**Fixed (`6b606a9`):** applied explicit `tickCount={5}` to the numeric axes of CenterComparisonChart, CenterDetailPanel, DistributionCharts (×3), ClinicalParametersRow, OutcomesPanel, IntervalHistogram, and ResponderView (single + cohort panels). Intentional `XAxis hide` preserved.
**Open (needs you):** please send the **original screenshot** so I can confirm the exact chart you flagged is among these and renders correctly.

### #9b — AUDIT-02: login rows showed `unauthenticated` instead of the username  ✅ Fixed (`6be5649`)
**Your finding (2026-05-27):** audit rows like `13:05:20  unauthenticated  Anmeldung … 200` — are these failed logins, and shouldn't the attempted username be logged?
**Investigated:** those `200` rows are *successful* logins (e.g. `admin`); failures would be `401`. The username **was** captured (in the `body`: `{"username":"admin","password":"[REDACTED]"}`) but the **actor** column fell back to `unauthenticated` because a login request has no token yet (`req.auth` empty). `unauthenticated` was never a username I chose — just the fallback label.
**Fixed:** `POST /api/auth/login` rows are now attributed to the attempted username (`auditMiddleware.ts`, AUDIT-02). Authenticated identity still wins (no spoofing); missing username → `unauthenticated`; `/verify` (username in the token, not the body) unchanged. Password stays redacted. 4 regression tests added; 1089/1089 green.
**Effect:** the audit log now shows *who* logged in and *which account* was targeted on a failed attempt — useful for brute-force/credential-stuffing monitoring. Safe (audit is admin-only; public 401 stays generic, no enumeration).

### #9 — PROT-001 deleted-user audit actor  🔎 Resolved by analysis (no fix)
**Found:** there is no deleted-user→anonymous path. Deleting a user calls `revokeByUsername()` synchronously (`userAdminApi.ts:172-179`) so their still-valid JWTs are dead; audit rows are append-only/immutable and retain the real username; the only fallback (now `'unauthenticated'`) is for 401/unauthenticated requests. A deleted user therefore cannot appear as anonymous.
**Your check:** confirm the live audit log shows no stray `anonymous`/deleted-user entries (if you do, capture one — it would indicate a path the analysis missed).
