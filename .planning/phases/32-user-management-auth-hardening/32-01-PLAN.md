<objective>
Add a user activation lifecycle and harden the admin user-management dialogs.

Backend first: introduce an `active` flag on `UserRecord` with a startup migration so existing
`users.json` records are treated as active; gate login against inactive users server-side; and
extend `PUT /api/auth/users/:username` to set `active`, revoking the target's sessions when
deactivated. Then the UI: add edit-dialog field/center validation to match create, and wire an
activation checkbox to the new `active` field.

Purpose: Closes UMGMT-01 (edit-dialog center enforcement), UMGMT-02 (all fields mandatory in both
dialogs), and UMGMT-03 (activation lifecycle with server-authoritative login rejection + immediate
session revocation).

Output: Migrated user model, login gate, admin de/reactivate control, validated create+edit dialogs,
and DE+EN i18n keys with passing tests.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| unauthenticated client → POST /login, /verify | attacker-controlled credentials; must not leak account state |
| admin client → PUT /api/auth/users/:username | privileged mutation crossing into the user store + session DB |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-01 | Information disclosure | POST /login inactive gate | mitigate | Inactive users return the identical generic 401 `{ error: 'Invalid credentials' }` as bad-credentials — no distinct "account disabled" message (preserves T-02-05 non-enumeration). |
| T-32-02 | Elevation of privilege | Held challenge token after deactivation | mitigate | Apply the inactive gate at POST /verify too, so a challenge token issued before deactivation cannot complete login. |
| T-32-03 | Spoofing / session persistence | Deactivated user's live refresh sessions | mitigate | revokeByUsername(target) on true→false transition immediately invalidates all refresh sessions (mirrors PROT-001 delete). |
| T-32-04 | Tampering | Client-supplied `active` value type | mitigate | Server validates `active` is a boolean before persisting; non-boolean → 400. Server is authoritative; client checkbox is convenience only. |
| T-32-05 | Repudiation | Activation changes | accept | PUT /users/:username already flows through auditMiddleware as a privileged POST/PUT event; no extra audit work in scope. |
</threat_model>
