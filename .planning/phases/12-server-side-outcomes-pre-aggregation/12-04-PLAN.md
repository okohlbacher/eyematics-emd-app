Request body:
```typescript
{
  cohortId: string,
  axisMode: 'days' | 'treatments',

  eye: 'od' | 'os' | 'combined',

  includePerPatient?: boolean,
```

Response (200):
```typescript
{
  median: GridPoint[],
  iqrLow: number[],
  iqrHigh: number[],
  perPatient?: PatientSeries[],
  scatter?: ScatterPoint[],
  meta: {
    patientCount: number,
    excludedCount: number,
    measurementCount: number,
    cacheHit: boolean,
  },
}
```

Error responses: 400 (invalid body), 401 (no auth), 403 (cohort not owned), 413 (oversized body), 502 (upstream data unavailable).

## TrajectoryResult shape (from shared/cohortTrajectory.ts)

The client CURRENTLY binds `aggregate` to `TrajectoryResult = { od: PanelResult; os: PanelResult; combined: PanelResult }`. The server response uses the D-03 projected shape (single eye + iqrLow/iqrHigh + meta). Plan 12-04 must either:
(a) Issue ONE server POST per eye (od, os, combined) and reassemble TrajectoryResult client-side, OR
(b) Keep the existing TrajectoryResult binding for <threshold cohorts and switch the panel consumers to read from the flattened server shape for >threshold cohorts.

Recommended: (a) — issue three parallel POSTs (one per eye) and reassemble into `TrajectoryResult`. This keeps the three `OutcomesPanel` components unchanged. Three POSTs share the same cache user-scope (D-08), so after the first call the other two are fast. The parallel fetch pattern preserves the "combined" panel's separate compute path that shared/cohortTrajectory already handles.
