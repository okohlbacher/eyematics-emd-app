Create server/authMiddleware.ts with:

1. Export `AuthPayload` interface:
```typescript
export interface AuthPayload {
  sub: string;
  preferred_username: string;
  role: string;
  centers: string[];
  iat: number;
  exp: number;
}
```

2. Extend Express Request type globally:
```typescript
declare global {
  namespace Express {
    interface Request {
      auth?: AuthPayload;
    }
  }
}
```

3. Export `createAuthMiddleware(jwtSecret: string)` factory function that returns Express middleware:
   - Define `PUBLIC_PATHS = new Set(['/api/auth/login', '/api/auth/verify', '/api/auth/config'])` (per D-08)
   - If `req.path` is in PUBLIC_PATHS, call `next()` and return
   - Extract `Authorization` header; if missing or not `Bearer ...`, respond 401 `{ error: 'Authentication required' }`
   - Call `jwt.verify(token, jwtSecret)` with type assertion to `AuthPayload`
   - On success: set `req.auth = payload`, call `next()`
   - On catch: respond 401 `{ error: 'Invalid or expired token' }`
   - Per AUTH-03: Structure supports future Keycloak by keeping verification logic swappable (the factory pattern makes this easy)

4. Use `import jwt from 'jsonwebtoken'` and `import type { Request, Response, NextFunction } from 'express'`

Per D-08, AUTH-01, AUTH-04. The middleware validates JWT on ALL /api/* routes except the three public auth endpoints. It extracts { username (from sub), role, centers } and attaches to req.auth for downstream handlers.

Create server/authApi.ts as an Express Router with three endpoints. Import `express`, `bcrypt` from `bcryptjs`, `jwt` from `jsonwebtoken`, `fs`, `path`, `crypto` from `node:crypto`.

**Settings integration:**
- Export `initAuth(settings: Record&lt;string, unknown&gt;, dataDir: string)` function that:
  - Reads `settings.auth.jwtSecret`; if absent, generates `crypto.randomBytes(32).toString('hex')` and writes back to settings.yaml (per D-07, Pattern 5 from RESEARCH.md)
  - Stores jwtSecret, maxLoginAttempts (default 5), twoFactorEnabled, and dataDir in module-level variables
  - Stores the fixed OTP code from settings.yaml `auth.otpCode` (default `'123456'`) for demonstrator use
  - Reads `settings.auth.defaultPassword` (default `'changeme2025!'`) for user migration
  - **CRITICAL (Pitfall 5):** Reads data/users.json, checks if first user has `passwordHash` field. If not, iterates all users and adds `passwordHash: bcrypt.hashSync(defaultPassword, 12)`. Writes back to file. Logs migration message.
- Export `getJwtSecret(): string` so Plan 03 can pass it to `createAuthMiddleware()`

**Failed login limiting (D-09, USER-13):**
- In-memory `Map<string, { count: number; lockedUntil: number }>` (per Pattern 4 from RESEARCH.md)
- `checkLock(username, maxAttempts)`: returns `{ locked, retryAfterMs }`. If `lockedUntil > Date.now()`, locked. If count >= max, compute backoff `2^count * 1000` ms, set lockedUntil, return locked.
- `recordFailure(username)`: increment count
- `resetAttempts(username)`: delete from map

**POST /api/auth/login (D-01 Step 1, USER-07, AUTH-02):**
- Parse body with `express.json()` middleware on the router (or use req.body if available)
- Actually, use `req.body` (express.json will be mounted globally in Plan 03). Fallback: parse manually with readBody if req.body is undefined.
- Extract `{ username, password }` from body
- `checkLock(username, maxLoginAttempts)` — if locked, respond 429 `{ error: 'Account temporarily locked', retryAfterMs }`
- Read data/users.json, find user by username (case-insensitive)
- If not found: respond 401 `{ error: 'Invalid credentials' }` (do NOT reveal user not found)
- `await bcrypt.compare(password, user.passwordHash)` — if false: `recordFailure(username)`, respond 401 `{ error: 'Invalid credentials' }`
- If twoFactorEnabled === false: `resetAttempts(username)`, sign JWT per D-06:
  ```
  jwt.sign({ sub: username, preferred_username: username, role: user.role, centers: user.centers }, jwtSecret, { algorithm: 'HS256', expiresIn: '10m' })
  ```
  Respond 200 `{ token }`
- If twoFactorEnabled === true: sign challenge token with purpose claim and 5-minute expiry:
  ```
  jwt.sign({ sub: username, purpose: 'challenge' }, jwtSecret, { algorithm: 'HS256', expiresIn: '5m' })
  ```
  Respond 200 `{ challengeToken }` (per Pitfall 7: purpose claim prevents misuse as auth token)

**POST /api/auth/verify (D-01 Step 2):**
- Extract `{ challengeToken, otp }` from body
- Verify challengeToken with jwt.verify; reject if expired or missing `purpose === 'challenge'`
- Extract username from `payload.sub`
- Compare otp against the fixed OTP code from settings
- If valid: `resetAttempts(username)`, read user from users.json, sign full JWT (same as login direct path), respond 200 `{ token }`
- If invalid: respond 401 `{ error: 'Invalid OTP' }`

**GET /api/auth/config (D-02):**
- No auth required (in PUBLIC_PATHS)
- Respond 200 `{ twoFactorEnabled }`

Export `authApiRouter` (the Express Router).

Per AUTH-05: The settings.yaml auth section will be extended by initAuth() to include jwtSecret (auto-generated), maxLoginAttempts (default 5), and the existing twoFactorEnabled is read from root level.

Per AUTH-09: JWT payload is `{ sub, preferred_username, role, centers, iat, exp }` — identical format for future Keycloak tokens.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> /api/auth/login | Untrusted username/password from browser |
| client -> /api/* | JWT Bearer token from sessionStorage |
| server -> data/users.json | Password hashes in file storage |
| server -> settings.yaml | JWT secret in config file |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | /api/auth/login | mitigate | bcrypt.compare for timing-safe password validation; never reveal if user exists vs wrong password |
| T-02-02 | Tampering | JWT token | mitigate | HS256 signature verification via jsonwebtoken; 10-min expiry |
| T-02-03 | Repudiation | login attempts | mitigate | Server-side audit middleware (Plan 02) logs all requests |
| T-02-04 | Info Disclosure | error messages | mitigate | Generic "Invalid credentials" for both user-not-found and wrong-password |
| T-02-05 | DoS | /api/auth/login | mitigate | Failed login limiting: 5 attempts, exponential backoff per D-09 |
| T-02-06 | Elevation | challenge token | mitigate | Challenge token has `purpose: 'challenge'` claim; authMiddleware rejects tokens with purpose field (Pitfall 7) |
| T-02-07 | Info Disclosure | settings.yaml jwtSecret | accept | settings.yaml is in public/ (committed); research notes this risk. Acceptable for demonstrator. Production would use settings.local.yaml or env override. |
</threat_model>
