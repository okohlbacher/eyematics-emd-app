Create server/auditDb.ts. Import `Database` from `better-sqlite3`, `path` from `node:path`, `crypto` from `node:crypto`.

**Define AuditDbRow interface:**
```typescript
export interface AuditDbRow {
  id: string;
  timestamp: string;
  method: string;
  path: string;
  user: string;
  status: number;
  duration_ms: number;
  body: string | null;
  query: string | null;
}
```

**Module-level state:** `let db: Database.Database;`

**Export `initAuditDb(dataDir: string): void`:**
- Resolve `path.join(dataDir, 'audit.db')`
- `db = new Database(dbPath)`
- `db.pragma('journal_mode = WAL')` (per research — concurrent reads during writes)
- Execute CREATE TABLE IF NOT EXISTS and CREATE INDEX IF NOT EXISTS using the D-14 schema exactly as shown in the interfaces block above
- Log `[audit] Initialized SQLite database at ${dbPath}`

**Export `logAuditEntry(entry: AuditDbRow): void`:**
- Use `db.prepare()` with INSERT INTO audit_log with named parameters (@id, @timestamp, etc.)
- Call `stmt.run(entry)`
- This is synchronous (better-sqlite3 is sync) — no async needed
- Generate `id` using `crypto.randomUUID()` if not provided

**Export `queryAuditEntries(filters: { user?: string; path?: string; from?: string; to?: string; limit?: number; offset?: number }): AuditDbRow[]`:**
- Build WHERE clause dynamically from provided filters
- `user` filter: `WHERE user = ?`
- `path` filter: `WHERE path LIKE ?` (use `%${path}%` for substring match)
- `from` filter: `WHERE timestamp >= ?`
- `to` filter: `WHERE timestamp <= ?`
- Default limit: 500, default offset: 0
- ORDER BY timestamp DESC
- Use `db.prepare(sql).all(...params)` with proper parameter binding
- Return typed array

**Export `exportAllEntries(): AuditDbRow[]`:**
- `db.prepare('SELECT * FROM audit_log ORDER BY timestamp DESC').all()` — no limit
- For admin full dump (AUDIT-03)

**Export `purgeOldEntries(): number`:**
- `const result = db.prepare("DELETE FROM audit_log WHERE timestamp < datetime('now', '-90 days')").run()`
- Log `[audit] Purged ${result.changes} entries older than 90 days`
- Return `result.changes`

**Export `startPurgeInterval(): NodeJS.Timeout`:**
- Call `purgeOldEntries()` immediately (on startup per D-15, AUDIT-08)
- `setInterval(() => purgeOldEntries(), 24 * 60 * 60 * 1000)` (daily)
- Return the interval handle

Per AUDIT-04 (SQLite at data/audit.db), AUDIT-01 (server-side only writes), AUDIT-08 (90-day retention), AUDIT-09/D-14 (schema — D-14 wins).

**Create server/auditMiddleware.ts:**

Import `logAuditEntry` from `./auditDb.js`, `crypto` from `node:crypto`, and Express types.

Export `createAuditMiddleware()` factory that returns Express middleware:
- Capture `startMs = Date.now()` at request entry
- Register `res.on('finish', () => { ... })` callback (per Pattern 2 from RESEARCH.md — do NOT override res.json):
  - `const duration = Date.now() - startMs`
  - `const user = req.auth?.preferred_username ?? 'anonymous'` (reads from authMiddleware's req.auth)
  - Per D-10: Log EVERY API request with method, path, user, timestamp, status, duration
  - Per D-11: For mutations (POST/PUT/DELETE), capture `JSON.stringify(req.body)` as body field (max 10KB, truncate if larger). For GET, capture `JSON.stringify(req.query)` as query field.
  - Call `logAuditEntry({ id: crypto.randomUUID(), timestamp: new Date().toISOString(), method: req.method, path: req.path, user, status: res.statusCode, duration_ms: duration, body, query })`
  - Wrap in try/catch — audit failure must NOT crash the request
- Call `next()` synchronously (the logging happens asynchronously on finish)

**Create server/auditApi.ts:**

Import `express`, `queryAuditEntries`, `exportAllEntries` from `./auditDb.js`.

Export `auditApiRouter` as Express Router:

**GET /api/audit (AUDIT-02):**
- Requires authentication (handled by authMiddleware upstream)
- Read query params: `user`, `path`, `from`, `to`, `limit` (default 500), `offset` (default 0)
- Parse limit/offset as integers with bounds (limit max 1000, offset min 0)
- Call `queryAuditEntries({ user, path, from, to, limit, offset })`
- Respond 200 with `{ entries, total: entries.length }`

**GET /api/audit/export (AUDIT-03):**
- Requires admin role: check `req.auth?.role !== 'admin'` — respond 403 `{ error: 'Admin access required' }`
- Call `exportAllEntries()`
- Set `Content-Disposition: attachment; filename="audit-export.json"` header
- Respond 200 with JSON array

**AUDIT-05 compliance:** No POST, PUT, PATCH, or DELETE routes are defined on this router. Only GET endpoints exist.

Per D-13: Only GET /api/audit and GET /api/audit/export. All writes are internal via logAuditEntry().

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> GET /api/audit | Authenticated user reads audit log |
| client -> GET /api/audit/export | Admin-only full dump |
| server middleware -> SQLite | Internal write path — no client control |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-08 | Tampering | audit_log table | mitigate | No write/delete API endpoints (AUDIT-05); only server-side logAuditEntry() can insert; only purgeOldEntries() can delete |
| T-02-09 | Info Disclosure | GET /api/audit | mitigate | Requires valid JWT; mutation bodies may contain sensitive data but are only visible to authenticated users |
| T-02-10 | Info Disclosure | GET /api/audit/export | mitigate | Admin role check; Content-Disposition forces download |
| T-02-11 | DoS | audit writes | accept | Synchronous better-sqlite3 writes are fast (~0.1ms); WAL mode prevents read blocking; acceptable for demonstrator load |
| T-02-12 | Repudiation | audit entries | mitigate | Every API request auto-logged by middleware; user cannot avoid creating audit trail |
</threat_model>
