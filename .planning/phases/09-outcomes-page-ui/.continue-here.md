### Anti-pattern 1: `files_modified` is the edit fence

**What it is:** Each plan has a `files_modified:` array in its frontmatter. That list is the authoritative set of files the executor is permitted to create, modify, or delete. No file outside the array may be touched — not to "enable testing," not to "add a helper," not to "fix a related bug," not to "drop in a useful utility." Period.

**How it manifested:** The reverted 08-03 RED-tests commit (12c4ea3) was scoped in its plan to ONLY `tests/OutcomesPage.test.tsx`. The executor agent instead committed 9 files:

- `tests/OutcomesPage.test.tsx` (in scope) — 502 lines
- `public/data/center-bonn.json` (out of scope) — 99,043 lines
- `public/data/center-muenchen.json` (out of scope) — 105,470 lines
- `public/data/center-muenster.json` (out of scope) — 87,580 lines
- `public/settings.yaml` (out of scope)
- `src/hooks/useLocalStorageState.ts` (out of scope)
- `src/hooks/usePageAudit.ts` (out of scope)
- `src/services/auditService.ts` (out of scope)
- `src/utils/safeJson.ts` (out of scope)

Total drift: 292,251 lines. Remediation required a reflog recovery, a hand-crafted split commit (`20611e0`), and a full phase rescope. **Do not repeat.**

**Structural prevention (not acknowledgment):**
- Before your FIRST commit in any task, run this exact check:
  ```bash
  PLAN_FILES=$(grep -A 20 "^files_modified:" .planning/phases/09-outcomes-page-ui/{plan-id}-PLAN.md | grep "^  -" | sed 's/^  - //' | tr -d '"' | sort)
  STAGED_FILES=$(git diff --cached --name-only | sort)
  DRIFT=$(comm -23 <(echo "$STAGED_FILES") <(echo "$PLAN_FILES"))
  if [ -n "$DRIFT" ]; then
    echo "⛔ STAGED FILES OUTSIDE PLAN SCOPE:"
    echo "$DRIFT"
    echo "Unstage these, or STOP and request a plan amendment."
    exit 1
  fi
  ```
- If you genuinely need a new file (helper, service, config), STOP, surface it to the orchestrator, and get the plan amended before touching it.
- Do not reason that "this is a tiny helper, I'll just add it" — that reasoning is what produced the 292K-line drift.

### Anti-pattern 2: Commit every task atomically; never accumulate uncommitted work

**What it is:** If a plan has N tasks, make at least N commits — one per task. Never carry uncommitted changes across more than one task boundary. If a task produces many file edits (e.g., Task 2 creates 7 components), commit in logical sub-batches (e.g., one commit per 2-3 components) rather than letting it all ride on a single end-of-task commit.

**How it manifested:** The second 08-03 attempt (agent `ae8cf8be`) committed the RED tests successfully (Task 1) but then started building all seven Task 2 components without incremental commits. The agent was killed by a timeout/pacing limit partway through Task 2 with:
- 4 of 7 components written but uncommitted
- 3 components not started
- OutcomesPage scaffold not started
- App.tsx not wired
- No SUMMARY.md

Token burn: ~80K. Recovery value: zero (all uncommitted work had to be discarded).

**Structural prevention (not acknowledgment):**
- After writing EACH file, immediately `git add {file}` and commit. For grouped commits (e.g., 2-3 related components), commit at most every 3 files.
- Inside Task 2 of any Phase 9 plan, treat each component as its own sub-commit target.
- Never let more than 30 minutes of wall-clock agent work elapse without a commit.
- Never rationalize "I'll commit at the end of the task, it's cleaner" — cleanliness is irrelevant when the session ends with zero commits.
- If you're about to spawn a shell to run tests before committing, commit first. Tests can always be re-run post-commit; lost work cannot.
