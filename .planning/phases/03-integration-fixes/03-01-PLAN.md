**Addresses review concern: _capturedBody type safety (Gemini MEDIUM + Codex MEDIUM consensus)**

Both Gemini and Codex flagged that using `(req as unknown as Record<string, unknown>)._capturedBody` is ad-hoc and type-unsafe. The fix is to extend the Express Request interface with a proper declaration.

**Create server/types.d.ts** with Express Request augmentation:

```typescript
/**
 * Express Request type augmentation for custom properties.
 *
 * _capturedBody: Raw body string captured by readBody() in server/utils.ts.
 * Used by auditMiddleware as a fallback when req.body is not populated
 * (i.e., for non-auth mutation routes where express.json() is not mounted).
 */
import 'express';

declare module 'express-serve-static-core' {
  interface Request {
    /** Raw body string captured by readBody() for audit middleware consumption. */
    _capturedBody?: string;
  }
}
```

**Why `express-serve-static-core`:** Express re-exports types from this package. Augmenting `express-serve-static-core` ensures `req._capturedBody` is recognized on all Express Request objects, including those in middleware signatures.

Verify that the project's tsconfig.json includes the `server/` directory (or does not exclude it) so the augmentation is picked up.

**Bug:** `auditMiddleware` reads `req.body` at `res.on('finish')` time (line 114), but `req.body` is only populated by `express.json()` which is scoped to `/api/auth` (server/index.ts line 144). For `POST /api/issues` and `PUT /api/settings`, the body is consumed by `readBody()` in the route handler, so `req.body` is undefined and audit entries get `body: null`.

**Addresses review concern: _capturedBody type safety (consensus).** With server/types.d.ts in place (Task 1), access `req._capturedBody` directly without `as unknown as Record` casts.

**Fix -- Part A (server/utils.ts):** Modify `readBody()` to attach the parsed body string to the request object after stream consumption. In the `req.on('end', ...)` callback, before calling `resolve(data)`, add:

```typescript
// Attach raw body to req for audit middleware capture (Bug 1 fix)
// Type augmentation in server/types.d.ts makes this type-safe
(req as import('express').Request)._capturedBody = data;
```

The `req` parameter is typed as `import('http').IncomingMessage` in readBody(). Since Express Request extends IncomingMessage, the cast to `import('express').Request` is safe and gives us access to the augmented `_capturedBody` property.

**Fix -- Part B (server/auditMiddleware.ts):** In the `res.on('finish')` callback, change the body capture logic (currently line 114-115) to fall back to `_capturedBody` when `req.body` is not populated.

Replace:
```typescript
const bodyStr =
  req.method !== 'GET' ? redactBody(urlPath, req.body) : null;
```

With:
```typescript
// Addresses review concern: clear rules for JSON vs YAML body handling
// Priority: req.body (from express.json()) > _capturedBody (from readBody())
// _capturedBody is always a raw string. tryParseJson attempts JSON parse;
// if it fails (e.g., YAML settings body), the raw string is kept as-is.
// redactBody handles both objects (from JSON.parse) and strings (raw YAML).
const rawBody = req.method !== 'GET'
  ? (req.body !== undefined && req.body !== null
      ? req.body
      : (req._capturedBody ? tryParseJson(req._capturedBody) : undefined))
  : undefined;
const bodyStr = req.method !== 'GET' ? redactBody(urlPath, rawBody) : null;
```

Note: `req._capturedBody` is accessed directly (no cast needed) because the type augmentation in server/types.d.ts adds it to the Express Request interface.

Also add a `tryParseJson` helper function above the middleware function:

```typescript
/**
 * Try to parse a string as JSON; return the original string if parsing fails.
 *
 * Rule: JSON bodies (POST /api/issues) get parsed into objects so redactBody
 * can traverse REDACT_PATHS. Non-JSON bodies (PUT /api/settings sends YAML)
 * stay as raw strings -- redactBody handles strings via String(body).
 */
function tryParseJson(str: string): unknown {
  try {
    return JSON.parse(str);
  } catch {
    return str;
  }
}
```

**Body handling contract (addresses Codex concern about tryParseJson rules):**
1. `req.body` (from express.json()) -- already a parsed object. Used for /api/auth routes.
2. `req._capturedBody` (from readBody()) -- always a raw string. Used for non-auth mutation routes.
3. `tryParseJson` attempts to parse the raw string as JSON. If successful, redactBody receives an object and can walk REDACT_PATHS. If parsing fails (YAML, malformed), redactBody receives the raw string and handles it via its fallback path.
4. `redactBody` already handles both objects (via JSON.stringify with path redaction) and non-objects (via String(body)). No changes needed to redactBody.

**Bug:** `validateSettingsSchema()` checks `obj.twoFactorEnabled` (top-level boolean) but the canonical settings.yaml structure has `auth.twoFactorEnabled` (nested). This means a valid settings.yaml with the nested auth section would be rejected because top-level `twoFactorEnabled` is missing.

**Fix:** Rewrite `validateSettingsSchema()` and add `export` keyword. Replace the entire function (lines 26-56 of settingsApi.ts) with:

```typescript
export function validateSettingsSchema(parsed: unknown): string | null {
  if (parsed === null || typeof parsed !== 'object') {
    return 'Settings must be a YAML object';
  }
  const obj = parsed as Record<string, unknown>;

  // Top-level numeric fields
  if (typeof obj.therapyInterrupterDays !== 'number' || !Number.isFinite(obj.therapyInterrupterDays)) {
    return 'therapyInterrupterDays must be a number';
  }
  if (typeof obj.therapyBreakerDays !== 'number' || !Number.isFinite(obj.therapyBreakerDays)) {
    return 'therapyBreakerDays must be a number';
  }

  // dataSource section
  if (obj.dataSource === null || typeof obj.dataSource !== 'object') {
    return 'dataSource must be an object';
  }
  const ds = obj.dataSource as Record<string, unknown>;
  if (typeof ds.type !== 'string' || ds.type.length === 0) return 'dataSource.type must be a non-empty string';
  if (typeof ds.blazeUrl !== 'string' || ds.blazeUrl.length === 0) return 'dataSource.blazeUrl must be a non-empty string';

  // auth section (canonical nested structure -- per D-03)
  if (obj.auth === null || typeof obj.auth !== 'object') {
    return 'auth must be an object';
  }
  const auth = obj.auth as Record<string, unknown>;
  if (typeof auth.twoFactorEnabled !== 'boolean') {
    return 'auth.twoFactorEnabled must be a boolean';
  }
  if (typeof auth.maxLoginAttempts !== 'number' || !Number.isInteger(auth.maxLoginAttempts) || auth.maxLoginAttempts < 1) {
    return 'auth.maxLoginAttempts must be a positive integer';
  }
  // otpCode is optional (defaults to '123456' in initAuth.ts)
  if (auth.otpCode !== undefined && typeof auth.otpCode !== 'string') {
    return 'auth.otpCode must be a string if provided';
  }

  return null;
}
```

- **REMOVED:** `obj.twoFactorEnabled` check (top-level -- wrong path)
- **ADDED:** `obj.auth` object existence check
- **ADDED:** `auth.twoFactorEnabled` boolean check (nested -- correct path)
- **ADDED:** `auth.maxLoginAttempts` positive integer check
- **ADDED:** `auth.otpCode` optional string check
- **ADDED:** `export` keyword so tests can import it directly (per D-05)
- **NOT ADDED:** `jwtSecret` -- JWT secret lives in `data/jwt-secret.txt`, NOT in settings.yaml (per RESEARCH.md pitfall 4)
