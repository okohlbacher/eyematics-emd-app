<objective>
Expose a thin, authenticated, no-op `GET /api/audit/events/view-open` endpoint so that
opening the Outcomes page produces one auditable row per view-open event.

Purpose: OUTCOME-11 and D-32 require a server-side audit entry when the Outcomes view
is opened. The audit log in this codebase is WRITE-ONLY from middleware (no POST API);
creating a new no-op GET causes `auditMiddleware.ts` to record the request with full
query string (which carries `name=open_outcomes_view` + the cohort identifier). This
is the established pattern in this codebase — see RESEARCH.md § Audit integration
and pitfall note "No POST to audit".

Output: Endpoint + unit tests. No client code here; the client call lives in 08-03.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Spoofing / Information Disclosure | `GET /api/audit/events/view-open` | mitigate | Endpoint is mounted inside `auditApiRouter` which is protected by `authMiddleware` (`server/index.ts` L198; authMiddleware mounted L~190). Unauthenticated requests return 401 before reaching the handler. `req.auth.preferred_username` is the identity recorded into `audit_log.user` by the middleware — clients cannot spoof the user field. Query string is user-controlled free-form; that is intentional (the client declares which cohort was opened), and the field is recorded as-is. No DB write occurs from this handler. |
| T-08-02 | Tampering | audit_log integrity | accept | The audit middleware inserts one row per request; tampering with the row post-write would require DB access, which is outside this endpoint's threat surface (covered by Phase 5 center-authz + DB filesystem permissions). |
| T-08-03 | Denial of Service | beacon spam | accept | Existing global rate limiter (`server/rateLimiting.ts`) applies to `/api/*`. No per-route limit added — the endpoint is cheap (204, no DB access from the handler itself) and the audit-log write is the same weight as any other authenticated GET. If abuse surfaces, apply `rateLimit({ max: 10, windowMs: 60_000 })` locally in a follow-up. |

The handler itself **introduces no new attack surface beyond what was already present
for `/api/audit`**: same auth, same middleware chain, same audit-write path. The only
incremental surface is the semantic meaning of the new URL, and that's covered above.
</threat_model>
