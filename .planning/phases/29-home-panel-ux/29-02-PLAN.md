<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| signed-in user → browser localStorage | Recent-activity records (route paths + pseudonyms) are written to a shared-device persistence layer. |
| user A ↔ user B on a shared workstation | Two clinicians may use the same browser profile sequentially. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-03 | Information Disclosure | recentActivityStore.ts (cross-user leakage) | mitigate | Per-username key `emd-recent:<username>` (D-01) — user B's hook reads only their own key; user A's case/pseudonym trail is never surfaced to user B. Asserted by the per-username isolation test (Plan 01). |
| T-29-04 | Information Disclosure | RecentActivityEntry contents | mitigate | Record shape stores only `id` (caseId), `label` (pseudonym — already visible in UI), `sub`/`path` (app-internal routes). NO clinical values, diagnoses, or tokens. `path` is an internal route, not an exfiltration vector. |
| T-29-05 | Denial of Service | localStorage.setItem (quota/private mode throw) | mitigate | Every localStorage call is try/catch-guarded; a throw is swallowed and the feature degrades to empty state — no crash. Asserted by the silent-failure test. |
| T-29-06 | Information Disclosure | residual trail (handled in Plan 04) | transfer | clearAll() and clear(username) are provided here; the logout wiring that calls them (D-02) lives in Plan 04. |
</threat_model>
