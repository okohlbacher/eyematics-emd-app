<objective>
Implement SESS-03: OAuth2-style refresh token rotation with RFC 6819 §5.2.2.3 family revocation. Every issued refresh JWT carries a server-generated `jti`; every `/api/auth/refresh` call looks up the jti in `sessions.db`, revokes the old row, inserts a new one, and emits a new cookie. Reuse of a revoked jti revokes the entire session family and forces re-login.

Purpose: Detect stolen-token replay and limit blast radius to one session family.
Output: Updated `jwtUtil.ts` (RefreshPayload + jti); updated `authApi.ts` (login inserts row, /refresh rotates + detects reuse); `tests/sessionRotation.test.ts` turned green; `tests/authRefresh.test.ts` updated for sessionsDb-aware setup.
</objective>

|-----------|----------|-----------|-------------|-----------------|
| T-27-03-01 | S (Spoofing) | stolen refresh token replay | mitigate | RFC 6819 §5.2.2.3 family revocation in /refresh handler: revoked or unknown jti → `revokeFamily(payload.sid)` + 401. Test `tests/sessionRotation.test.ts` asserts all sibling rows transition to revoked=1 on replay. |
| T-27-03-02 | T (Tampering) | client edits payload.iat to bypass absolute cap | mitigate | Absolute cap check uses `existing.issued_at` from the server-authoritative sessions row, NOT `payload.iat`. Tampered iat is irrelevant. |
| T-27-03-03 | T (Tampering) | race between lookup, revoke, insert | mitigate | better-sqlite3 sync API + WAL serialization (inherited from Plan 02). No async gap; entire rotation block executes synchronously inside the Express handler. |
| T-27-03-04 | E (Elevation) | pre-Phase-27 token without jti accepted | mitigate | Empty-jti sentinel from jwtUtil → `getSession('')` returns null → family revocation path → 401. D-18 documents this is intentional re-login. |

| T-27-03-06 | I (Disclosure) | error message leaks server internals | mitigate | Reuse-detected response is exactly `{ error: 'Refresh token reuse detected' }`; cap response is `{ error: 'Session cap exceeded' }`. No stack traces, no jti values exposed to client. |
