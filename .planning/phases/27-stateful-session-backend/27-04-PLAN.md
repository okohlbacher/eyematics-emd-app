<objective>
Implement SESS-04: admin-triggered refresh-token signing-key rotation with a dual-key verification window. Operators stage the next key in `data/jwt-secret-next.txt`, then call `POST /api/auth/rotate-key` (admin only). The server atomically renames files (next→current, current→prev), reloads in-memory secrets, and reports when the prev key fully expires. Tokens signed by the prev key continue to verify until they hit the 12h absolute cap, then gracefully return 401 (never 500).

Purpose: Allow operators to rotate the JWT signing key without forcing immediate re-login for all users.
Output: Updated `initAuth.ts` (dual-key state + rotate function); updated `jwtUtil.ts` (dual-key verify); new admin endpoint in `authApi.ts`; `tests/rotateKey.test.ts` green.

```typescript
export function rotateSigningKey(): { rotatedAt: string; prevKeyExpiresBy: string } {
  if (!_dataDir || !_settings) throw new Error('[initAuth] rotateSigningKey() called before initAuth()');

  const currentPath = path.join(_dataDir, 'jwt-secret.txt');
  const nextPath    = path.join(_dataDir, 'jwt-secret-next.txt');
  const prevPath    = path.join(_dataDir, 'jwt-secret-prev.txt');

  if (!fs.existsSync(nextPath)) {
    const err = new Error('jwt-secret-next.txt not found in data directory');
    (err as Error & { code?: string }).code = 'NEXT_KEY_MISSING';
    throw err;
  }

  // If a prev exists from a recent rotation, delete it (the previous window has ended or operator accepts overlap)
  if (fs.existsSync(prevPath)) fs.unlinkSync(prevPath);

  // Atomic-ish rename pair
  fs.renameSync(currentPath, prevPath);
  fs.renameSync(nextPath, currentPath);

  try { fs.chmodSync(currentPath, 0o600); } catch { /* best effort */ }
  try { fs.chmodSync(prevPath, 0o600); } catch { /* best effort */ }

  // Reload in-memory state to match disk
  _jwtSecretPrev = fs.readFileSync(prevPath, 'utf8').trim();
  _jwtSecretCurrent = fs.readFileSync(currentPath, 'utf8').trim();

  const rotatedAt = new Date().toISOString();
  const prevKeyExpiresBy = new Date(Date.now() + _settings.refreshAbsoluteCapMs).toISOString();
  console.log(`[initAuth] signing key rotated at ${rotatedAt}; prev key valid until ${prevKeyExpiresBy}`);
  return { rotatedAt, prevKeyExpiresBy };
}
```

1. **`server/jwtUtil.ts`** — update `verifyRefreshToken` to dual-key:

```typescript
import jwt from 'jsonwebtoken';
import { getJwtSecrets } from './initAuth.js';

export function verifyRefreshToken(token: string): RefreshPayload {
  const { current, prev } = getJwtSecrets();
  let payload: unknown;
  try {
    payload = jwt.verify(token, current, { algorithms: ALGS });
  } catch (err) {
    // Pitfall 2: only fall through on wrong-key (JsonWebTokenError), NOT on expiry
    if (err instanceof jwt.JsonWebTokenError && !(err instanceof jwt.TokenExpiredError) && prev) {
      payload = jwt.verify(token, prev, { algorithms: ALGS });
    } else {
      throw err;
    }
  }
  const p = payload as RefreshPayload;
  if (p.typ !== 'refresh') throw new Error('wrong_token_type');
  // Plan 03 sentinel: empty jti is allowed here; /refresh handler treats it as family-revocation trigger
  if (typeof p.jti !== 'string') (p as RefreshPayload).jti = '';
  return p;
}
```

| Boundary | Description |
|----------|-------------|
| admin client → POST /api/auth/rotate-key | authenticated admin Bearer; CSRF-protected per existing pattern |
| filesystem → initAuth | jwt-secret-*.txt files at chmod 0o600; readable only by server process user |
| client cookie → /refresh during window | refresh JWT may have been signed by current OR prev key |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-04-01 | E (Elevation) | `/rotate-key` called by non-admin | mitigate | Handler checks `req.auth.role === 'admin'`, returns 403. Endpoint NOT in PUBLIC_PATHS — unauthenticated calls hit authMiddleware → 401. Both paths tested in tests/rotateKey.test.ts. |
| T-27-04-02 | T (Tampering) | attacker writes a malicious jwt-secret-next.txt | accept | Requires write access to `data/` — already privileged (same trust level as the server process). Out of threat model for v1 (single-host deployment). Filesystem ACLs are the operator's responsibility. |
| T-27-04-03 | S (Spoofing) | attacker who stole prev key forges tokens during window | mitigate | Dual-key window is bounded by `refreshAbsoluteCapMs` (12h). Operator should rotate again if compromise suspected — second rotation deletes the compromised prev key. |
| T-27-04-04 | D (DoS) | expired-prev token causes 500 / crash | mitigate | Pitfall 2: TokenExpiredError does NOT fall through to prev key; expired tokens hit standard 401 path. Pitfall 6: initAuth re-reads disk state on restart so server boots cleanly mid-window. Tested by "401 once absolute cap elapses (not 500)" test. |
| T-27-04-05 | I (Disclosure) | error response leaks signing key bytes | mitigate | Handler returns only `{ rotatedAt, prevKeyExpiresBy }` on success and generic error strings on failure. Console logs print only timestamps and key_id hash prefixes — never raw key material. |
| T-27-04-06 | T (Tampering) | concurrent /rotate-key calls race on rename | accept | Two concurrent admin rotations is operator-error scenario; second `fs.renameSync` of a missing source throws, handler returns 500. Acceptable for v1; admin UI in Phase 28 will gate this. |
| T-27-04-07 | S (Spoofing) | attacker forges TokenExpiredError-shaped token to bypass prev fallback | mitigate | Verifier discriminates on `instanceof jwt.JsonWebTokenError && !(instanceof TokenExpiredError)` — both checks come from the jsonwebtoken library's verified error hierarchy, which is signature-validated.
