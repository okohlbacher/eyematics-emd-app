<objective>
Relabel the audit actor fallback from `'anonymous'` to `'unauthenticated'` for
requests that are rejected (401) or that arrive with no JWT. This is a
security/audit-correctness fix (PROT-001 / AUDIT-01): `'anonymous'` implies a
known identity that chose not to log in; `'unauthenticated'` accurately signals
that no identity was established.

Purpose: Ensure audit log consumers (SIEM, compliance review) can distinguish
unauthenticated probes from deliberate anonymous access patterns. The change is
strictly additive — no existing rows are touched, and authenticated actors are
unaffected.

Output:
- `server/auditMiddleware.ts` — fallback string updated, comment updated
- `server/auditDb.ts` — schema DEFAULT updated (fresh DB only)
- `server/auditApi.ts` — view-open handler fallback updated (same pattern)
- `server/index.ts` — inline mount comment updated
- `tests/auditMiddleware.test.ts` — existing test description + new assertions
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP request → auditMiddleware | Untrusted; actor identity derived from JWT claim, not request header |
| auditMiddleware → auditDb | Internal; actor string is application-controlled, not user-supplied |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-38-01 | Repudiation | auditMiddleware actor fallback | mitigate | `'unauthenticated'` is a sentinel value distinguishable from real usernames; no request can produce an empty or null actor (NOT NULL DEFAULT + application fallback both enforce presence) |
| T-38-02 | Spoofing | JWT preferred_username claim | accept | JWT is already validated by authMiddleware (HMAC-signed, expiry checked); this plan does not change auth logic — the claim is trusted post-verification |
| T-38-03 | Tampering | Audit row immutability | accept | No UPDATE or DELETE is introduced; append-only semantics preserved; DEFAULT change applies only to INSERT of fresh rows; no migration of existing rows |
| T-38-04 | Information Disclosure | Actor label masking authenticated users | mitigate | Authenticated path (`req.auth?.preferred_username`) is evaluated first; `'unauthenticated'` is only reached when `req.auth` is undefined — an authenticated actor can never be mislabeled as 'unauthenticated' |

ASVS L1 alignment: V7.3 (log integrity — no row suppression or modification introduced); V2.1 (authentication state correctly reflected in audit — unauthenticated probe vs. logged-in user now distinguishable).
</threat_model>
