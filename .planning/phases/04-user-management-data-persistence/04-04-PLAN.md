<objective>
Fix the auth token flow so the client authenticates via the server and uses real JWTs.

Purpose: Plans 04-01/02/03 built correct server endpoints and correct client wiring, but the login flow still validates credentials client-side using hardcoded DEFAULT_CREDENTIALS and getAuthHeaders() sends a Base64-encoded JSON object instead of a real JWT. This plan rewires the two load-bearing files (AuthContext.tsx and authHeaders.ts) plus LoginPage.tsx to complete the client-to-server auth bridge.

Output: Working end-to-end authentication where login POSTs to server, JWT is stored, and all API calls succeed.
</objective>

<interfaces>
<!-- Server login API contract (from server/authApi.ts) -->

POST /api/auth/login
  Request body: { username: string, password: string }
  Response (2FA disabled): { token: string }        -- full session JWT
  Response (2FA enabled):  { challengeToken: string } -- short-lived challenge token
  Error 401: { error: 'Invalid credentials' }
  Error 429: { error: 'Account locked', retryAfterMs: number }

POST /api/auth/verify
  Request body: { challengeToken: string, otp: string }
  Response: { token: string }                        -- full session JWT
  Error 401: { error: 'Invalid or expired challenge token' } | { error: 'Invalid OTP' }
  Error 429: { error: 'Account locked', retryAfterMs: number }

JWT payload shape (from signSessionToken):
  { sub: string, preferred_username: string, role: string, centers: string[], iat: number, exp: number }

GET /api/auth/config (public, no auth required)
  Response: { twoFactorEnabled: boolean }

<!-- authHeaders contract (target state) -->
sessionStorage key: 'emd-token' stores raw JWT string
getAuthHeaders() returns { Authorization: 'Bearer <jwt>' } or {}

<!-- AuthContext.login() contract (target state) -->
login(username, password, otp?) returns Promise<{ ok: boolean; error?: string }>
  - Async (currently sync) -- LoginPage must await it
  - On success: stores JWT in sessionStorage('emd-token'), sets user from JWT payload, returns { ok: true }
  - On failure: returns { ok: false, error: 'invalid_credentials' | 'account_locked' | 'invalid_otp' | 'network_error' }

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04-01 | Spoofing | sessionStorage JWT | mitigate | JWT is server-signed with HS256 + server secret; cannot be forged client-side. Token expires in 10 min (signSessionToken expiresIn: '10m'). authMiddleware.ts calls jwt.verify() on every request. |
| T-04-04-02 | Information Disclosure | JWT payload in sessionStorage | accept | JWT payload contains username, role, centers (no secrets). sessionStorage is per-tab, cleared on tab close. Acceptable risk for SPA session. |
| T-04-04-03 | Tampering | JWT payload modification | mitigate | Server verifies JWT signature via jwt.verify(). Modified payload invalidates signature -> 401. Client decodes payload only for display (setUser), never for authorization decisions. |
| T-04-04-04 | Information Disclosure | DEFAULT_CREDENTIALS in client bundle | mitigate | This plan deletes DEFAULT_CREDENTIALS, VALID_OTP, and DEFAULT_MANAGED_USERS from AuthContext.tsx. No hardcoded credentials remain in client code. |
| T-04-04-05 | Elevation of Privilege | Client-decoded role from JWT used for access | accept | Client uses role only for UI display (show/hide admin tabs). All authorization is enforced server-side by authMiddleware checking JWT. Client role is cosmetic. |
| T-04-04-06 | Spoofing | challengeToken replay in 2FA flow | mitigate | challengeToken has 2-min expiry and purpose='challenge' claim. Server verifies both before issuing session JWT. |
</threat_model>
