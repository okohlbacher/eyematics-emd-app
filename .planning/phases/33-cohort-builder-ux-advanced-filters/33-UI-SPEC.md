## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| xs | 4px | `gap-1` / `p-1` | Icon gaps, inline padding, badge padding |
| sm | 8px | `gap-2` / `p-2` | Element gap within a field group |
| md | 16px | `gap-4` / `p-4` | Default element spacing, section margins |
| lg | 24px | `gap-6` / `p-6` | Panel internal padding (matches existing filter panel `p-5`) |
| xl | 32px | `gap-8` / `p-8` | Layout gaps between major sections |
| 2xl | 48px | `gap-12` | Major section breaks |
| 3xl | 64px | `gap-16` | Page-level spacing |

### Spacing Exceptions

The following non-standard values are brownfield carry-overs. They must not be changed without addressing visual drift in the existing UI.

| Value | Tailwind | Context | Justification |
|-------|----------|---------|---------------|
| 6px vertical padding | `py-1.5` | Filter field inputs (age min/max, CRT min/max, HbA1c min/max) | Brownfield carry-over — matches existing CohortBuilderPage inputs (RESEARCH lines 296–309, CohortBuilderPage.tsx lines 445, 462); changing to `py-2` would increase input height relative to all existing adjacent controls and create visual drift. Do not change. |
| 4px top margin | `mt-1` | Validation error `<p>` elements | Matches the `saveName` validation block at CohortBuilderPage line 596; standard xs token. |

Touch targets: all clickable controls minimum 36px height (8px vertical padding on `text-sm` = 32px + border = ~34px; preset buttons use `size="sm"` from `Button` which is `py-1 text-xs` = adequate for desktop clinical use).

## Surface 1: Cohort Builder Filter Input Validation (COH-01)

**Primary focal point:** The live results count (e.g. "14 cases of 328 total") is the primary focal point of the Cohort Builder page — it updates in real time as valid filters change. All filter controls are secondary to this count.

### Interaction Contract

**Trigger:** On every `onChange` event for age min, age max, Visus min text, Visus max text, CRT min, CRT max.

**Validation rules:**

| Field | Rule | Error key |
|-------|------|-----------|
| Age min | `isNaN(value)` OR `value < 0` | `cohortValidationAgeNonNumeric` |
| Age max | `isNaN(value)` OR `value < 0` | `cohortValidationAgeNonNumeric` |
| Age range | `ageMin > ageMax` (both set and valid) | `cohortValidationAgeLowerExceedsUpper` |
| Visus min | Parsed float `> 1` | `cohortValidationVisusOutOfRange` |
| Visus max | Parsed float `> 1` | `cohortValidationVisusOutOfRange` |
| Visus range | `visusMin > visusMax` (both set and valid) | `cohortValidationVisusLowerExceedsUpper` |
| CRT min | `isNaN(value)` OR `value < 0` | `cohortValidationCrtNonNumeric` |
| CRT max | `isNaN(value)` OR `value < 0` | `cohortValidationCrtNonNumeric` |
| CRT range | `crtMin > crtMax` (both set and valid) | `cohortValidationCrtLowerExceedsUpper` |

**Error element spec** (mirrors `saveName` validation block, CohortBuilderPage lines 594–603):

```
<p
  id="{fieldId}-validation"
  role="alert"
  className="mt-1 text-xs px-2 py-1.5 rounded border
             bg-red-50 border-red-200 text-red-600
             dark:bg-[--color-coral-soft] dark:border-[--color-coral] dark:text-[--color-coral-ink]"
>
  {t(errorKey)}
</p>
```

Note: `py-1.5` on the error element is a brownfield carry-over (see Spacing Exceptions table).

- Placement: directly below the input row (min–max pair), NOT below each individual input.
- One error element per field group (age, Visus, CRT) — shows the first applicable error for that group.
- `role="alert"` for all field validation errors (all are hard errors that block Save).
- The invalid field is NOT applied to the live results list (D-03). The filter engine ignores the invalid field; results update from remaining valid filters.

**Save button disabled state:**
- `disabled={hasAnyFilterError || !saveName.trim() || hasHardError}` where `hasAnyFilterError` is true when any of the nine rules above fires.
- Existing `disabled:opacity-50 disabled:cursor-not-allowed` classes on the `button` element (CohortBuilderPage line 585) already render the disabled affordance correctly — no change needed.
- No tooltip needed on the disabled button; the inline error messages are sufficient.

## Surface 2: Filter Persistence & Reset (COH-02)

### Interaction Contract

**Persistence:** Filter state is stored in `sessionStorage` under key `emd-cohort-filters` (JSON). No visual cue is shown when filters are restored — the filter inputs simply reflect the stored values on mount. This is consistent with how other persisted state (recent activity, theme) operates — silent restoration, no toast or banner.

**Rationale for no "filters restored" cue:** The filter inputs are visible in the sidebar; the user sees their state immediately. A "filters restored" banner would add noise with no action path.

**Reset control** (existing at CohortBuilderPage line 558–563):
- Keep existing layout: full-width button below the CRT range group.
- Keep existing classes: `flex-1 py-2 text-sm border border-gray-200 dark:border-gray-700 rounded-lg hover:bg-gray-50 dark:hover:bg-gray-700/50 dark:text-gray-300`.
- On click: `setFilters({})` + `sessionStorage.removeItem('emd-cohort-filters')` + clear `visusMinText`/`visusMaxText`.
- No confirmation dialog needed — Reset is non-destructive (no data is deleted).

## Surface 3: Issue-Based Preset Buttons (COH-03)

### Decision: Visual Treatment

**Preset buttons are rendered as a labeled button group in a dedicated section above the existing filter criteria** in the filter panel sidebar. Placement: between the panel heading (`filterCriteria`) and the Diagnosis checkboxes.

Rationale: Presets are the fastest path to a meaningful cohort — placing them at the top of the filter panel ensures discoverability without burying them. They are clearly separated from the manual filters by a section divider and label, preventing confusion between "I clicked a preset" and "I set manual filters."

**Visual layout:**

```
[section label: "cohortPresets" — text-xs font-semibold uppercase tracking-wide text-ink-3]
[2×2 button grid, gap-2]
  [Therapie-Abbrecher]    [Unplausible CRT-Werte]
  [Flagged data quality]  [Implausible Visus]
[divider: border-t border-[var(--color-line)] mt-3 mb-4]
```

**Button element spec for each preset:**

Default (no preset active):
```
<Button
  variant="ghost"
  size="sm"
  onClick={() => applyPreset('therapyBreaker')}
  aria-pressed="false"
>
  {t('presetTherapyBreaker')}
</Button>
```

Active (this preset is the current `filters.preset`):
```
<Button
  variant="soft"
  size="sm"
  onClick={() => clearPreset()}
  aria-pressed="true"
  className="ring-1 ring-[var(--color-teal)]"
>
  {t('presetTherapyBreaker')}
</Button>
```

- `Button variant="ghost"` default state: `bg-[var(--color-surface)] border-[var(--color-line-2)]` — neutral, clearly secondary to the Save CTA.
- `Button variant="soft"` + `ring-1 ring-[var(--color-teal)]` active state: soft background signals "something is applied here"; teal ring distinguishes from a plain soft button.
- `size="sm"`: `px-3 py-1 text-xs rounded-md` — compact, fits a 2-column grid in the 4-column sidebar. (`px-3` = 12px; comfortable touch target at this text size.)
- `aria-pressed`: communicates toggle state to screen readers.
- Clicking the active preset again clears `filters.preset` (toggle behavior).
- Clicking any manual filter field after applying a preset clears `filters.preset` (D-RESEARCH). The button returns to default state.

**Grid layout class:** `grid grid-cols-2 gap-2 mb-3`

## Surface 4: Advanced Filter Dialog (COH-04)

### Decision: Modal vs Expandable Panel

**Decision: Centered modal overlay.**

Rationale:
1. The filter sidebar is already at `col-span-4` width (~280–320px). Adding 5 more filter fields inline would overflow the panel and require scroll within a scrollable region (poor UX).
2. The existing precedent for complex editing in this app is the centered modal: `QualityFlagDialog` (`fixed inset-0 bg-black/30 flex items-center justify-center z-50` — QualityFlagDialog.tsx line 36), `FeedbackButton` modal (same pattern). A slide-in drawer is used for settings/compare (right-edge panels), but those panels have indefinite content; the advanced filter set is bounded (exactly 5 attributes).
3. A modal maintains focus discipline (focus trap), which matters for clinical accuracy — the user must confirm or cancel the advanced filters as a unit.
4. The 5 curated attributes (diagnosis subtype, comorbidities, HbA1c, drug/agent, laterality) can fit comfortably in a `max-w-lg` modal without scrolling.
5. An expandable panel below the existing filters would push the Save section off-screen on smaller viewports, breaking the spatial relationship between filter inputs and the Save action.

**Entry affordance:**

A "Advanced filters" trigger button below the Reset button, above the Save section divider. Uses `Button variant="ghost" size="sm"` with a `Sliders` icon. When any advanced attribute is set, the button shows `Button variant="accent" size="sm"` to signal active state.

```
<Button
  variant={hasAdvancedFilters ? 'accent' : 'ghost'}
  size="sm"
  icon={<Sliders className="w-3.5 h-3.5" />}
  onClick={() => setShowAdvancedDialog(true)}
>
  {t('advancedFilters')}
  {hasAdvancedFilters && <Badge tone="teal" className="ml-1">{activeAdvancedCount}</Badge>}
</Button>
```

`hasAdvancedFilters`: true when any of `diagnosisSubtype`, `comorbidities`, `hba1cRange`, `medicationCodes`, `laterality` is non-empty.
`activeAdvancedCount`: count of non-empty advanced fields (integer, shown in the Badge when > 0).

### Modal Spec

**Container:**
```
fixed inset-0 bg-black/30 flex items-center justify-center z-50
```

**Panel:**
```
bg-white dark:bg-gray-800 rounded-xl shadow-xl p-6 w-full max-w-lg
```

Max-width `max-w-lg` (512px) — wider than `QualityFlagDialog`'s `max-w-md` (448px) to accommodate the 5 fields at comfortable widths.

**Header row:** `text-base font-semibold text-gray-900 dark:text-white` heading + `X` button (Lucide `X` icon, `p-1 rounded hover:bg-gray-100`, `aria-label={t('advancedFiltersDiscard')}`). Same pattern as `OutcomesSettingsDrawer` line 99–109.

**Field specifications:**

| Attribute | Control type | Options / Range | CohortFilter key |
|-----------|-------------|-----------------|-----------------|
| Diagnosis subtype | Multi-select checkboxes (same style as existing Diagnosis checkboxes) | AMD subtypes + DR subtypes from the SNOMED code vocabulary in `shared/fhirCodes.ts` | `diagnosisSubtype?: string[]` |
| Comorbidities | Single checkbox: "Has any comorbidity" (boolean) | On / off | `hasComorbidity?: boolean` |
| HbA1c | Two numeric inputs (Min / Max) with `inputMode="numeric"` | 0–20 % (mmol/mol range 0–219) — rendered with unit label "%" | `hba1cRange?: [number, number]` |
| Drug / agent | Multi-select checkboxes | Medication codes from the app's medication vocabulary (resolve at plan time from generated bundle inspection) | `medicationCodes?: string[]` |
| Laterality | Three radio buttons: OD (right eye), OS (left eye), OU (both) | Single-select | `laterality?: 'OD' \| 'OS' \| 'OU'` |

HbA1c inputs use `px-2 py-1.5` (brownfield carry-over — see Spacing Exceptions table) to match all other filter field inputs.

**Rationale for comorbidities as boolean:** RESEARCH.md Open Question 2 flags that the multi-select approach requires enumerating SNOMED codes from generated bundles. The boolean "has any comorbidity" is unambiguous, avoids a long code list in the UI, and covers the primary clinical use case (filtering to patients WITH vs WITHOUT comorbidities). The planner may upgrade to multi-select after bundle inspection, but the UI contract specifies boolean for now.

**Laterality naming:** OD/OS/OU are the standard ophthalmologic abbreviations; the DE/EN labels include the full form in parentheses. See i18n keys below.

**Combination semantics:** All advanced attributes combine with AND (matches existing filter behavior throughout the app). No OR toggle is exposed. This is consistent with CONTEXT D-DEFER (OR-logic deferred).

**Footer:**

```
<div className="flex gap-2 justify-end pt-4 border-t border-gray-100 dark:border-gray-700 mt-4">
  <Button variant="ghost" size="sm" onClick={handleClear}>
    {t('advancedFiltersClear')}
  </Button>
  <Button variant="ghost" size="sm" onClick={onClose}>
    {t('advancedFiltersDiscard')}
  </Button>
  <Button variant="accent" size="sm" onClick={handleApply}>
    {t('advancedFiltersApply')}
  </Button>
</div>
```

- **Apply filters** (`advancedFiltersApply`): writes the dialog's local state to `filters` and closes the modal. Triggers sessionStorage write via existing `useEffect`.
- **Clear fields** (`advancedFiltersClear`): resets all five advanced fields to empty within the dialog (does not close).
- **Discard changes** (`advancedFiltersDiscard`): closes the modal, discards any changes made since opening. Also used as `aria-label` on the `X` icon-only close button in the header.
- No destructive confirmation needed — Clear only affects the dialog's temporary state; Apply is required to commit.

**Accessibility:**
- `role="dialog"` on the panel.
- `aria-modal="true"`.
- `aria-labelledby` pointing to the heading element.
- `X` close button: `aria-label={t('advancedFiltersDiscard')}` (icon-only button must have an accessible label).
- Focus trap on open (same pattern as `FeedbackButton.tsx` lines 40–60).
- Escape key closes the modal (same pattern as `OutcomesSettingsDrawer` / `CohortCompareDrawer` Escape listener).
