## 🔵 Trajectory (Verläufe)

### N1 — Line hover should emphasize along the WHOLE line, not only at measurement points
Tester: "highlighting should work whenever I hover over a line, not only when I hit an actual measurement."
**Diagnosis:** Plotly `hovermode:'closest'` fires `plotly_hover` only near a data point. To emphasize anywhere along a line we need line-proximity hovering — e.g. `hovermode:'closest'` won't do it; options: add `hoveron:'points+fills'` (lines don't support fills), or widen the hit area, or switch the per-patient lines to a mode that reports hover along the path. Practically: increase hover tolerance / use `hovermode:'closest'` with invisible wider hit-lines, OR accept Plotly's point-based hover. **Investigate feasibility; if the framework can't hover mid-segment cheaply, document.**
**Confidence:** medium (Plotly constraint).

### N2 — Scatter points should be highlighted on hover (not just tooltip)
Tester: "Scatter plots should also be highlighted when I hover over them; right now there are tooltips only, no highlighting."
**Diagnosis:** the scattergl hover shows the tooltip but doesn't emphasize the hovered marker. Add an imperative highlight on `plotly_hover` over the scatter — e.g. an overlaid highlight marker (bigger/ringed) at the hovered point, restored on unhover (mirrors the line-emphasis approach).
**Confidence:** high.

### N9 — Y-axis metric must be self-evident without opening settings (new)
Tester: "it must be self-evident what y-metric settings are used without opening the settings. Add a y-axis label."
**Diagnosis:** the Plotly panels currently have no visible y-axis title; add an axis title reflecting the active metric (logMAR / Δ logMAR / Δ% · CRT µm / Δ / Δ%).
**Confidence:** high.

## 🔵 Cohort overlay — IOD axis + distribution plots

### N4 — IOD relative axis still shows individual dates for some patients
Tester: "For some patients the IOD axis does switch, but the individual dates are not removed → fix (e.g. EM-UKT-0027)."
**Diagnosis:** when the IOD axis switches to relative months, residual calendar-date ticks/data remain for some patients (likely the patient IOP series still keyed on `date` while the axis is `relMonths`, or duplicate rows). Ensure the IOD relative axis fully replaces dates (patient series re-keyed to `relMonths`, no leftover date ticks). Repro on EM-UKT-0027.
**Confidence:** high (concrete repro case).

### N5 — Distribution (histogram) plots: patient bars not aligned to the % y-axis; rework when overlay is on (BIG)
Tester: "the patient bars are not aligned with the percentage y-axis, so patient vs cohort isn't comparable → useless. When the cohort overlay is active, change the bar plots to: (1) y-axis fully percentage; (2) one bar = patient % per measurement range; (3) one bar = cohort median % per measurement range; (4) tooltips with absolute counts (cohort median + patient actual); (5) add a tooltip explanation."
**Diagnosis:** the distribution charts currently plot patient counts and cohort % on mismatched scales, so the bars aren't comparable. Rework (overlay-on): both series as **percentage of their own total per bin**, grouped bars (patient % vs cohort-median %), a single % y-axis, and tooltips carrying the absolute counts + an explanation. (Overlay-off can stay as-is.)
**Confidence:** high; clear spec. Largest item this round.

## 🔴 Reversal — owner decision needed

### N8 — Vollzähligkeit denominator: tester now wants it windowed (reverses round-6)
Tester: "The denominator should be adapted to the registered total in the time range."
**Context:** in round 6 you (owner) explicitly chose **"count restricts, % keeps the full registered denominator"** (so a window lowers the % as coverage, e.g. 14%→~4%), specifically to avoid the % collapsing toward ~100%. The tester now asks for the **denominator to follow the range too** — which is what WS-D originally implemented and the review flagged (numerator≈denominator for activity-derived windows → ~100%). The tester's rationale is consistency (count and % should both reflect the range). **This directly reverses the round-6 decision → confirm with owner before changing.**
**Confidence:** the request is clear; the conflict with the prior decision is the issue.
