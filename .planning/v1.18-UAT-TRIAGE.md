## 🔵 Trajectory / Verläufe (Plotly) follow-ups

### M1 — Hovered per-patient line still not highlighted/emphasized (feasibility-gated)
Tester: "Hovered line is still not highlighted / emphasized. Only change when feasible in this framework."
**Diagnosis:** per-patient lines are individual Plotly traces (`perpatient-${id}`, plotlyTraces.ts:130-135). Hover-emphasis is feasible via `plotly_hover` → `Plotly.restyle` (bump that trace's width/opacity), but must be cheap (restyle one trace, not a full redraw). Attempt it; if it can't be made smooth, document why (per the tester's "only when feasible").
**Confidence:** medium (feasible; perf-sensitive).

### M2 — Scatter hover shows TWO tooltips; keep only the detailed one (NEW)
Tester: "now there are two tooltips, a) one with pseudonym only and b) one containing further information; keep b) only."
**Diagnosis:** the scattergl trace has `hovertemplate: '%{text}<extra></extra>'` (plotlyTraces.ts:167) → Plotly's **native** hover label (pseudonym), AND `plotly_hover` drives our **custom imperative** tooltip (the detailed one). Both render. Fix: set the scatter trace `hoverinfo:'none'` (kills the native label but the `plotly_hover` event still fires, so the custom detailed tooltip remains). Likewise drop the per-patient-line native `hovertemplate` if it double-shows.
**Confidence:** high.

### M3 — Compare mode: scatter-points toggle has no effect; scatter + lines should be opt-in, per-cohort coloured (NEW)
Tester: "There is a scatter-points option when comparing cohorts, which does not have an effect. Scatter points and individual lines should be opt-in; displayed in the respective colors."
**Diagnosis:** `buildCrossCohortTraces` (plotlyTraces.ts:176+) only emits the median + IQR per cohort (`hoverinfo:'skip'`), no scatter or per-patient lines — so the layer toggles are inert in cross mode. Add scatter (scattergl) and per-patient line traces per cohort, gated on the existing layer toggles, each in its cohort's colour.
**Confidence:** high.

### M6 — Calendar-date X-axis is not linear in time
Tester: "The axis for actual dates is not linear → should be fixed."
**Diagnosis:** the date axis uses `dataKey="date"` as a **category** axis (VisusCrtChart.tsx:250-251; same in baseline-change + IOD) → ticks are evenly spaced regardless of the real time gaps. Convert to a **time/linear** axis: numeric timestamp domain (`type="number"`, `scale="time"` or epoch-ms dataKey with a date `tickFormatter`), so spacing is proportional to elapsed time. Apply to all date-axis case-detail charts.
**Confidence:** high. Medium effort (touches each chart's XAxis + the data keying).

## 🔵 Dark mode

### M11 — Theme cycle shouldn't include a redundant step (NEW)
Tester: "when in system design (light) the cycle is on system design but it does not skip the light mode in the cycle (which it should)."
**Diagnosis:** the toggle cycles light → dark → system → light. When on **System** resolving to light, the next step (explicit Light) is visually identical — a no-op click. Make the cycle skip the step whose appearance equals the current effective theme (e.g. System-light → Dark directly).
**Confidence:** medium (clear intent; pick a sensible cycle rule).

### M12 — Some tables/plots still don't switch to dark mode
Tester: "Single tables and plots are still not changing to dark mode." (screenshot)
**Diagnosis:** L11b coverage is incomplete — specific tables/plots remain light. Audit the codebase for charts/tables NOT using the theme-aware colour helpers / missing `dark:` classes (likely doc-quality, admin, audit, landing, or distribution surfaces not touched in v1.17), and fix them. Confirm against the live app in dark mode.
**Confidence:** high it's incomplete; needs an audit to enumerate offenders.
