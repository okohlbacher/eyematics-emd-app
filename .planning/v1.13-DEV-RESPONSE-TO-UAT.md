## B1 — Time filter now scopes the documentation-quality metrics (Q2)

**You wrote:** "if I want to check whether my quality increased / decreased for the last 6 months, I should only see metrics that are based on values from that time range … the smaller the range, the less patients, the lower the Vollzähligkeit metric, which is not providing any information." Plus: "we should discuss whether we can include more time range options."

**What changed:**
- All Dokumentationsqualität / Datenqualität metrics (Vollständigkeit, Plausibilität, per-centre scores) are now computed **only from observations inside the selected time window**.
- Both the numerator **and the denominator** of completeness scope to the window — so a period that is well-documented reads ~100% regardless of older gaps, instead of collapsing as the window shrinks.
- Added a **3-month preset** and a **custom from/to date picker** (presets are now 3m / 6m / 1y / all + custom).

**How to verify:** Dokumentationsqualität → switch all → 6m → 3m: the Grundgesamtheit count drops as expected, but the score cards stay meaningful (they do **not** crater toward 0). Pick "Benutzerdefiniert" and set a historical window — the cards reflect only that range, including the upper bound.

**Status:** ✅ Done & verified (live-tested across time zones; cards reacted 245 → 9 → 169 patients with scores staying meaningful).

## B2 — "Vollzähligkeit" naming collision (#3)

**You wrote:** "Vollzähligkeit is computed different to the landing page."

**What changed:** the two metrics that shared the word "Vollzähligkeit" are now named distinctly, each with an ℹ definition tooltip:
- Landing page: **Einwilligungsquote** (share of consenting patients).
- Dokumentationsqualität: **Dokumentations-Vollständigkeit** (per-patient completeness of mandatory fields), and the per-observation metric is now **Messwert-Vollständigkeit**.

**How to verify:** compare the landing page tile and the Dokumentationsqualität cards — different names; hover the ℹ icons for the definitions.

**Status:** ✅ Done & verified.

## C2 — Quality-check selection moved to the Datenqualität tab (Additional point 1)

**You wrote:** "I do not think it is self-explanatory, why quality measures are … in the Kohortenbildung tab, and what they do. This should probably move to the Datenqualität tab."

**What changed:** the quality-check checklist is **removed from Kohortenbildung** and now lives on the **Datenqualität tab**, where it edits the selected cohort's checks **in place** and saves immediately (a short explanatory note says what the checks do). The selection still persists with the cohort.

**How to verify:** Datenqualität → pick a cohort → the "Qualitätsprüfungen für diese Kohorte" card appears; toggle a check → reload → the selection persists and the case anomalies honour it. Kohortenbildung no longer shows the checklist; saving a new cohort still works (defaults to all checks).

**Status:** ✅ Done & verified (server-side authorisation confirmed — you can only edit your own cohorts).

## C1 — Quality case-review reworked into one inline surface (#4)

**You wrote:** "interaction concept is questionable: we now have two options to do that. Any argument against confirming results directly in the „Messwerte" list? … overloaded page; Auffällige Werte / Fehlende Daten highlighted in its own card but not in the list. … „Behoben", but no option to edit data → not self explanatory."

**What changed (design-led rework):**
- **One confirmation path:** the duplicate top/bottom approve controls are gone. You now confirm **per measurement, inline in the list** (plus one bulk "alle unauffälligen als bestätigt" button). The old results card is now a read-only audit log.
- **Anomalies & missing values are inline:** anomalous rows are highlighted amber with an "auffällig" pill and a reason; missing parameters appear as their own "fehlt" rows — all in the list you review. The separate card is gone; its count remains as a "Nur auffällige" filter.
- **Status vocabulary with definitions:** Offen · bestätigt · auffällig · **Behoben = „im Quellsystem korrigiert"** (you fixed the value upstream in the source EHR; the demonstrator itself changes no data — clarified in a tooltip, with a distinct colour/icon from "bestätigt", and the stale value shown struck through).
- The page is condensed to a header strip + one review card + one collapsed log.

**How to verify:** Datenqualität → open a flagged case → there are two cards. Confirm a normal row inline (→ green "bestätigt", persists on reload). On an anomalous row use "Behoben" (→ blue, value struck through). The bulk button confirms only clean, unjudged rows. Missing Visus/CRT/IVOM show as "fehlt" rows. Hover the legend ℹ icons for definitions.

**Status:** ✅ Done & verified (a design mockup was reviewed and approved before build; inline confirm persists across reload).

## C3 — One-step "Kohorte aufteilen" (split) wizard (Additional point 2)

**You wrote:** "it might be easier to have different options here for the user: Split by gender, diagnosis, centers; split by a list of ranges for age, visus, retinal thickness."

**What changed:** a new **"Kohorte aufteilen"** action (next to the existing single-subcohort action on each saved cohort) creates **all** sub-cohorts in one step and opens compare mode:
- **Categorical:** split by gender / diagnosis / centre (one sub-cohort per value).
- **Range:** split by age / Visus / CRT, either with **your own cut points** or **automatic equal-size (quantile) groups**; a live preview shows each group and its case count before you confirm.

**How to verify:** Kohortenbildung → on a cohort, click the split icon → choose an attribute (e.g. Geschlecht → two groups; or Alter → quantile → three balanced groups) → review the preview counts → "Unterkohorten erstellen". The children appear as `Parent:Group` and feed the comparison view.

**Status:** ✅ Done & verified (categorical and range/quantile previews confirmed live).

## D1 — Multi-centre select on Dokumentationsqualität (#1, discussion)

**You wrote:** "there is no multi-center select in Dokumentationsqualität (Do we need that though? → Discussion)."

**Decision:** left as-is. Dokumentationsqualität is inherently a per-centre comparison view (it already breaks every metric down by centre), so a centre filter would be redundant there. The multi-centre filter remains on Analyse and Datenqualität.

**Status:** ⚪ No change (by decision).
