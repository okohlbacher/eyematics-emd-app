### I1 — A1: hover highlights the wrong point (MEDIUM)
Tester: "Scatter points are now clickable, but mouseover only highlights one datapoint that is on the same point on the x axis as the mouse, but not the scatter point hovered over."
**Diagnosis:** the ComposedChart uses a shared (axis-level) tooltip → hovering activates the nearest **x-category**, not the specific scatter point under the cursor. So the highlight/tooltip can attach to a different point that shares the x position.
**Proposed measure:** give the scatter its own item-level (non-shared) hover so the hovered point is the one highlighted — e.g. an `activeShape`/per-point hover on the Scatter, or a custom tooltip keyed to the hovered datum. Keep the drill-down click working.

### I2 — A1/A6: performance still poor + no loading status (HIGH, recurring)

**Diagnosis:** the Verläufe tab still blocks the main thread — likely the synchronous aggregation over 245 patients × thousands of observations and/or rendering thousands of scatter nodes (≈7k dots + 7k hit-halos when scatter is on). **Confidence:** high it's needed; the exact bottleneck needs a profiling spike first (don't guess-optimise).

## Adversarial review synthesis (internal code-reviewer + Codex 0.137, both code-grounded)

Both reviews agree; corrections to my initial triage:

- **I1 (hover) — escalate to correctness bug.** The shared (axis-level) tooltip pipeline drives BOTH the highlight AND the drill-down target (`OutcomesPanel.tsx:266,299`); the active point is "nearest x", not "under the cursor", so the **drill-down can navigate to the wrong patient** when points share an x-band. `Tooltip shared={false}`/`activeShape` won't work cleanly with the hand-drawn custom shape. **Fix:** track an explicit hovered-datum from the Scatter/custom-shape mouse events and make both the highlight and the click use that datum (euclidean-nearest), not the tooltip ref.
- **I2 (perf) — confirmed root cause.** Server-aggregation exists but its threshold defaults to **1000** (`settingsService.ts:52`), so the 245-case cohort runs **client-side with no spinner** (the loading status is gated to server mode, `OutcomesView.tsx:88`). Worse (Codex): `layers` initialises with scatter/per-patient **ON**, and the auto-off only fires in a post-first-render **effect** (`useOutcomesRouteState.ts:117,226,240`) — so the **first paint renders ~14k SVG circles** (7,130 points × 2 circles), then strips them. **Fix order:** (1) derive the large-cohort layer defaults **before first render** (no heavy first paint); (2) add a **client-side loading/status** indicator (not only server mode); (3) cap/downsample or canvas-render scatter when it is on. A web worker is secondary (only if profiling shows aggregation dominates on big cohorts).
- **I3a/I3b — cheap & confirmed** (CRT median + bands have `legendType="none"`; band is IQR p25–p75).
- **I3c — localised, but axis change is the invasive part.** The absolute-month binning is in the **case-detail overlay** (`useCaseData.ts:356`), NOT the shared Outcomes trajectory (which already does relative days, `cohortTrajectory.ts:501`). Two sub-options: **(i) relative-time AGGREGATION only** — bucket peers by months-since-their-own-baseline but keep emitting the reference on the patient's absolute date, so the **X-axis stays absolute** (cheap, clinically what NF wants); **(ii) full relative-time X-axis** — invasive (VisusCrtChart X=`date`, PatientHeader timeline keys on absolute date, `PatientHeader.tsx:206`).
- **I4 — confirmed.** Set the Visus line `name` to the full label (`VisusCrtChart.tsx:254`), shorten/remove the caption; watch legend wrap.
- **I5 (B2 reversal) — net-new computation, not a relabel.** Landing = `cases.length / countRawPatients(bundles)` (stub-inclusive, `LandingPage.tsx:65`, `fhirLoader.ts:89`). DocQuality has **no** such metric and **no per-centre raw (stub-inclusive) denominator** today — it only sees stub-excluded `cases`. Implementing "Vollzähligkeit = present/total" on DocQuality requires plumbing `bundles` grouped by centre. Both reviewers expect NF means **replace** the current completeness card (keep 4 cards). Plausibility & Overall cards currently lack tooltips → add 2.
- **I6a — REFUTED as data-loss; it is a display-gate bug.** The note (`errorType`) is preserved (status-only update, `DataContext.tsx:211`); it just stops rendering because the pill is gated to `status==='open'` (`QualityCaseDetail.tsx:485,510`). **Fix:** render non-sentinel `errorType` for acknowledged/resolved rows too (and in the audit log), suppressing the `confirmed`/`corrected` sentinels. Higher priority — erodes the already-accepted C1. **I6b:** add `title` to Bestätigen (it already has aria-label).
