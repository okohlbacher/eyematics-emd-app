### J2 — Verläufe performance STILL poor (I2 follow-up, HIGH, recurring)
Tester: "The 'wird berechnet' message shows very briefly, then **white cards**, and the dashboard **remains unresponsive until the plots are loaded**. Render the plots initially with a self-explanatory loading status **while keeping the UI responsive** (add/remove Anzeigeebenen, close Einstellungen). **The view is reset when leaving to another view.** **Can we use multithreading?**"
**Diagnosis:** the v1.14 two-pass mount removed the *first-paint* 14k-circle freeze and added a status, but the **actual plot render is still a synchronous main-thread block** — status flashes → white cards → freeze while Recharts builds the panels. The controls (layer toggles, settings) are unresponsive during that block. Also the trajectory view state (layers/metric) is **not persisted** across navigation → resets on return.
**Proposed measures (need a decision on depth):**
- Progressive / deferred panel rendering (mount panels one at a time / on idle) so the main thread yields and controls stay responsive; a persistent, self-explanatory loading state until done.
- Move heavy aggregation off the main thread (**Web Worker**) — NF explicitly asks. Feasible for `computeCohortTrajectory`; the client path currently runs it synchronously.
- Persist the trajectory view state (layers, metric, scatter) in the URL/store so leaving + returning doesn't reset it.
- Reconsider scatter rendering cost (canvas vs capped SVG) for the enabled case.
**Confidence:** high it's still broken; the approach + effort need an owner decision (worker + progressive render is the substantive path).
