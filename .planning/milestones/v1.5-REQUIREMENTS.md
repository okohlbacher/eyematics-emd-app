## v1.5 Requirements

### Site Roster (SITE-*)

- [ ] **SITE-01**: The configured site roster lists exactly 7 centers — Aachen, Chemnitz, Dresden, Greifswald, Leipzig, Mainz, Tübingen — in `data/centers.json` and in `server/constants.ts` `DEFAULT_CENTERS` fallback.
- [ ] **SITE-02**: The client-side `CENTER_SHORTHANDS` map in `src/services/fhirLoader.ts` matches the new 7-center roster with stable `org-*` IDs.
- [ ] **SITE-03**: Removed centers (Bonn, München, Münster) no longer appear in any loaded data — their `org-ukb`, `org-lmu`, `org-ukm` IDs are absent from config, users, bundles, and tests.
- [ ] **SITE-04**: `data/users.json` center assignments are migrated — removed-center IDs are stripped from each user's `centers` array; any user with an empty resulting array is reassigned to `org-uka`.
- [ ] **SITE-05**: Tests that pin the center list (`tests/constants.test.ts`, `tests/ui-requirements.test.ts`, `tests/fhirApi.test.ts`, `tests/dataApiCenter.test.ts`, `tests/centerBypass.test.ts`) are updated to assert the new 7-center roster and pass on CI.
- [ ] **SITE-06**: German and English UI references to centers (login hints, cohort filters, admin UI labels) reflect the new roster with no leftover Bonn/München/Münster strings.
- [ ] **SITE-07**: Documentation (`README.md`, `docs/Pflichtenheft.md`, `docs/Konfiguration.md`, `docs/Lastenheft.md`) describes the 7-site roster and lists the new center IDs consistently.

### Synthetic Data Generator (DATA-GEN-*)

- [ ] **DATA-GEN-01**: A Node.js generator script (`scripts/generate-center-bundle.ts`) produces a valid FHIR R4 Bundle for a given center id, shorthand, and patient count — structure matches existing `center-aachen.json` (Organization + Patient/Condition/Observation/Procedure/MedicationStatement/ImagingStudy entries).
- [ ] **DATA-GEN-02**: Generator supports the three clinical cohorts present in existing bundles — age-related macular degeneration (AMD), diabetic macular edema (DME), retinal vein occlusion (RVO) — with proportions configurable via CLI flags.
- [ ] **DATA-GEN-03**: Each generated patient has a longitudinal trajectory — baseline + follow-up visus observations (LOINC-coded) per eye, 1–20 IVOM procedures with aflibercept/bevacizumab medication statements, plausible CRT observations, and a realistic birthDate/gender distribution.
- [ ] **DATA-GEN-04**: Generator is deterministic given a seed — the same inputs produce byte-identical output, so bundles can be regenerated reproducibly.
- [ ] **DATA-GEN-05**: Generator exposes an npm script (`npm run generate-bundles`) that regenerates bundles for all 5 new sites at ~45 patients each, writing to `public/data/center-<city>.json`.
- [ ] **DATA-GEN-06**: Generated bundles satisfy the same schema expectations as hand-authored ones — `fhirLoader.ts` loads them without errors, cohort queries return the expected counts, and audit/data endpoints accept them.

### Cohort Outcome Trajectories (OUTCOME-*)

- [ ] **OUTCOME-01**: From any saved cohort (or an ad-hoc filtered cohort), the user can open an "Outcomes" view that scopes to that cohort's members.
- [ ] **OUTCOME-02**: The Outcomes view renders three chart panels — right eye (OD), left eye (OS), and combined OD+OS — each showing per-patient visus curves with a bolder median overlay.
- [ ] **OUTCOME-03**: The X axis supports two modes — "Days since baseline" (real time) and "Number of treatments" (treatment index) — toggled via the curve-settings panel.
- [ ] **OUTCOME-04**: The Y metric supports three modes — "Absolute" (logMAR or Snellen), "Delta" (Δ vs each patient's first measurement), and "Delta %" — toggled via the curve-settings panel.
- [ ] **OUTCOME-05**: Display toggles let the user independently show/hide the median line, per-patient curves, a scatter of all individual measurements, and an SD-shading band around the median.
- [ ] **OUTCOME-06**: The median (and SD band) is computed by interpolating each patient's curve onto a shared grid whose resolution is controlled by a user-visible slider (default 120 grid points, range 20–300).
- [ ] **OUTCOME-07**: Summary cards at the top of the view show cohort patient count, total measurement count, OD measurement count, and OS measurement count — all recomputed live when the cohort filter or toggles change.
- [ ] **OUTCOME-08**: An expandable "Data preview" panel exposes the underlying measurement rows (patient pseudonym, eye, days-since-baseline, treatment index, visus value) and supports CSV export.
- [ ] **OUTCOME-09**: Outcome aggregation runs entirely client-side over data the server has already authorized — no new data-bypass path; center-based data restriction still applies.
- [ ] **OUTCOME-10**: Median / SD / interpolation math lives in a pure utility (`src/utils/cohortTrajectory.ts`) with unit-test coverage exercising edge cases (empty cohort, single patient, single measurement, sparse series, mismatched patient spans).
- [ ] **OUTCOME-11**: Outcome view is audited — opening it generates a server-side audit entry (same pattern as other analytical views) including cohort identifier or filter snapshot.
- [ ] **OUTCOME-12**: UI is fully localized (de/en) matching the app's existing i18n convention — no untranslated strings in the curve-settings panel, display toggles, panel titles, or summary cards.

## Future Requirements (Deferred)

- Server-side pre-aggregation for very large cohorts (>1000 patients) — v1.5 computes client-side; a server endpoint can be added later if perf warrants.
- Additional outcome metrics beyond visus (CRT trajectory, treatment-interval distribution, responder classification) — scoped as future analytics features once OUTCOME-01..12 prove the pattern.
- Cross-cohort comparison on the same chart — current scope is single-cohort.

## Out of Scope (explicit)

- Changes to cohort definition/filtering semantics — the Outcomes view consumes cohorts the existing `CohortBuilderPage` already produces.
- Real patient data — synthetic generator produces pseudonymous data only; no PHI pathways are added.
- Statistical inference (p-values, significance testing) — the view is descriptive, not inferential.
- Export to external analysis tools beyond CSV (no SPSS/R/SAS formats).
