## Cross-Phase Integration

### Confirmed Wiring (23/25 requirement integration paths)

All Phase 1 and Phase 2 modules are correctly imported, mounted, and wired in `server/index.ts`. Middleware ordering is correct (audit before auth before routes). Frontend auth flow is complete from LoginPage through JWT storage to authenticated API calls.

### Integration Gaps (3 found)

**1. MEDIUM: Audit body capture for non-auth mutations (AUDIT-09, AUDIT-01)**

`auditMiddleware` reads `req.body` but `express.json()` is only mounted on `/api/auth`. For `POST /api/issues` and `PUT /api/settings`, the body is consumed by `readBody()` raw stream reader, so `req.body` is never populated. Audit entries for these routes have `body: null`.

**2. LOW: Time filter param name mismatch (AUDIT-02)**

`auditService.ts` sends `from`/`to` query params; `auditApi.ts` reads `fromTime`/`toTime`. Date filtering silently fails. Currently dormant — AuditPage doesn't use date filters yet.

**3. LOW: Settings schema validator mismatch (AUTH-05)**

`settingsApi.ts` `validateSettingsSchema` requires top-level `twoFactorEnabled`, but `settings.yaml` moved it to `auth.twoFactorEnabled`. A settings write-back round-trip could fail with 400 or corrupt the auth section.
