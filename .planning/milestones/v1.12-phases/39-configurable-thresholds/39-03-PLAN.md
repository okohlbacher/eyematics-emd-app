<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → POST /api/outcomes/aggregate | request body (cohortId etc.) crosses; centers come from req.auth, not body |
| settings.yaml → aggregation | operator-configured thresholds influence which cases enter a cohort |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-06 | Elevation of Privilege | resolveCohortCases center scoping | mitigate | Threshold options are injected ONLY into applyFilters; the existing center filter (req.auth.centers, request-body override ignored, isBypass guarded) is untouched — a threshold edit can never widen a cohort beyond the user's authorized centers. Verified the center-filtering lines remain unchanged. |
| T-39-07 | Tampering | aggregate cache poisoning across settings change | mitigate | Cache key now includes threshold values AND settingsApi PUT calls invalidateAllAggregates() — a post-change request cannot read a pre-change cached result, preventing a stale-bounds cohort from being served. |
| T-39-08 | Spoofing/bypass | client supplying its own threshold options | accept | The endpoint does not accept threshold options from the body — server derives them from settings.yaml only. No body field for thresholds is added, so a client cannot widen its own cohort by sending looser bounds. |
</threat_model>
