# Requirements: EMD Milestone v1.6 — Outcomes Polish & Scale

**Defined:** 2026-04-16
**Core Value:** Every user sees only authorized data, with tamper-proof audit trail — while `/outcomes` stays fast, visually polished, and useful beyond visus.

**Scope one-liner:** Close v1.5 visual QA, add server-side pre-aggregation for large cohorts, extend outcome metrics beyond visus, and fix Phase 9 code-review INFO findings.

## v1.6 Requirements

### Visual / UX QA (VQA)

Follow-through on the `human_needed` and visual-QA items flagged during v1.5 audit. Deliverables are observable UI behaviors, not just "looks nicer" claims.

- [ ] **VQA-01**: Admin filter UI on the user-management page renders all 7 sites (Aachen, Chemnitz, Dresden, Greifswald, Leipzig, Mainz, Tübingen) and filters correctly — verified by Playwright/RTL snapshot test locked to the 7-site roster.
- [ ] **VQA-02**: Outcomes chart palette meets WCAG AA contrast for median / per-patient / scatter series against panel background in both light and dark mode — codified by a visual-regression or contrast-unit test.
    - *Phase 10 scope note (2026-04-16):* Verified in **light mode only** (panel background `#ffffff`). Dark-mode contrast deferred — codebase has no dark-mode infrastructure (no Tailwind theme config, no `prefers-color-scheme`, no theme provider). Rationale and deferral captured in `.planning/phases/10-visual-ux-qa-preview-stability/10-CONTEXT.md` §Deferred. VQA-02 remains partially open pending dark-mode work in a future milestone.
- [ ] **VQA-03**: IQR band renders with a palette-consistent fill + stroke that remains distinguishable from median line at all zoom levels; fallback when IQR is empty (n<2 at grid point) renders no band, not a 0-height artifact.
- [ ] **VQA-04**: Tooltip on outcomes scatter + median layers shows patient id / eye / x-value / y-value with localized units; suppressed for per-patient series when that layer is off.
- [ ] **VQA-05**: Empty-state copy for `/outcomes` (no cohort selected, cohort with zero eligible measurements, cohort with all-filtered eyes) is distinct per case and localized DE + EN.

### Server-Side Pre-Aggregation (AGG)

Keep `/outcomes` responsive for cohorts >1000 patients by moving median/IQR/grid computation server-side. Must preserve v1.5's center-filtered security boundary and match client output byte-wise at the grid level.

- [ ] **AGG-01**: New `POST /api/outcomes/aggregate` endpoint accepts `{ cohortId, axisMode, yMetric, gridPoints, eye }` and returns `{ median, iqrLow, iqrHigh, perPatient?, scatter? }` — center-filtered from JWT, not from request body.
- [ ] **AGG-02**: Server aggregation uses the same algorithm as the client `aggregate` useMemo (Phase 8-01); parity verified by a round-trip test: seed a cohort, compute both ways, assert `median` + `iqrLow` + `iqrHigh` are byte-identical.
- [x] **AGG-03**: Client `/outcomes` auto-routes to the server endpoint when cohort size exceeds a configurable threshold (default 1000 patients, overridable via `settings.yaml`); below threshold keeps the current client path.
- [ ] **AGG-04**: Server aggregation response is cacheable per `{ cohortId, axisMode, yMetric, gridPoints, eye }` key with explicit cache invalidation on cohort mutation (ETag or in-memory TTL — chosen during plan phase).
- [ ] **AGG-05**: Aggregation endpoint emits an audit event `outcomes.aggregate` with cohort id hash (not raw id), user, center set, and payload size — PII-minimal per IN-01 guidance.

### New Outcome Metrics (METRIC)

Extend the outcomes view beyond visus. Each metric reuses the existing axis/y-metric/layers infrastructure.

- [ ] **METRIC-01**: CRT (Central Retinal Thickness) trajectory panel mirrors the visus panel contract: OD / OS / OD+OS, absolute + Δ + Δ%, median / per-patient / scatter / IQR layers, and share the axis-mode + grid-points controls.
- [ ] **METRIC-02**: Treatment-interval distribution view shows per-patient injection gaps (days between consecutive IVOM events) as a histogram + median line; filterable by eye.
- [ ] **METRIC-03**: Responder classification bucketizes cohort eyes into `responder` / `partial` / `non-responder` by a configurable threshold (default Δvisus ≥ 5 letters at 12 months) and shows bucket counts + median trajectory per bucket.
- [ ] **METRIC-04**: Metric selector on `/outcomes` lets the user switch between Visus / CRT / Interval / Responder without navigation; URL query param `metric=` deep-links to the selection.
- [ ] **METRIC-05**: CSV export works for every metric with metric-appropriate columns (e.g., CRT export replaces `va_logmar` with `crt_um`); filename carries the metric slug.
- [ ] **METRIC-06**: Full DE + EN i18n for all new metric strings (panel titles, metric names, tooltip labels, empty-state copy, CSV headers) with an i18n completeness test matching the v1.5 pattern.

### Phase 9 Code-Review Findings (CRREV)

- [ ] **CRREV-01** (IN-01): Audit beacon `GET /api/audit/events/view-open` no longer carries cohort id in querystring; cohort reference sent as hashed id in the event payload, verified by test against the audit DB row.
- [ ] **CRREV-02** (IN-02): `OutcomesDataPreview` rows use a stable composite key (patient id + eye + measurement timestamp) instead of array index — verified by a React key-uniqueness test when the same rows render in different orders.
