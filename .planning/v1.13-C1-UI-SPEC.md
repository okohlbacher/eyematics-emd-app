## 1. Current-state analysis (code-grounded)

All line refs are to `src/components/quality/QualityCaseDetail.tsx` unless noted.

### 1.1 The four cards rendered today (top → bottom)
1. **Case-info card** (lines 129–245): pseudonym + exclude badge; three action buttons (`Als geprüft markieren`, `Von Analysen ausschließen`, `Vollständige Fallansicht`); a 5-column meta grid (age / diagnosis / total measurements / center / therapy status). **Embeds approve-control #1** at lines 177–203.
2. **Anomalies + missing-data card** (lines 247–290): amber card, only rendered when `anomalies.length > 0`. Lists each anomaly/missing item with parameter, value, reason, and a `Fehler melden` button or a status pill.
3. **Tabs card — "Zu prüfende Messwerte" / OCT** (lines 293–375): the measurements table (lines 324–371) has columns Parameter / Datum / Wert / Aktion. The Aktion column is only a flag icon (line 358–364) or a status pill (line 346–356). **Anomalies are NOT marked on these rows.**
4. **Prüfergebnisse card** (lines 378–411): **approve-control #2** — the full list of flags again, each with a status `<select>`.

### 1.2 NF point 1 — two redundant approve controls (CONFIRMED)
- **Control #1** — `data-testid="top-flag-status-controls"` (lines 177–203). Added in v1.12 for QUAL-025 ("reachable near top without scrolling"). Renders one `<select>` (Offen/Bestätigt/Behoben) per flag.
- **Control #2** — Prüfergebnisse card (lines 378–411). Renders the *same* flags with the *same* `<select>` plus reporter/date metadata.
- Both call the identical `onUpdateFlagStatus`. They are duplicate editors of the same data → exactly NF's "two options to do that".
- There is a *third* touch-point: the per-row flag icon in the measurements table (line 358) that only **creates** a flag (opens the dialog) — it cannot set status. So a reviewer flags in one place and confirms in two others.

### 1.3 NF point 2 — anomalies only in a separate card (CONFIRMED)
The amber card (lines 247–290) is the only place "Auffällige Werte / Fehlende Daten" are surfaced. The measurements table (lines 334–369) renders every observation with no anomaly styling — the offending Visus/CRT rows look identical to normal ones. Missing data (no Visus / no CRT / no injections) has **no row at all** in the table (the table maps `selectedCase.observations`, and a missing parameter has no observation), so it is invisible in the list.

### 1.4 NF point 3 — "Behoben" unclear (CONFIRMED)
Status vocabulary today (`src/i18n/translations.ts`):
- `statusOpen` = "Offen" (open)
- `statusAcknowledged` = "Bestätigt" (acknowledged)
- `statusResolved` = "Behoben" (resolved)

Wire enum (`shared/types/fhir.ts:202`): `'open' | 'acknowledged' | 'resolved'`. None carries a tooltip/definition. "Behoben" with no edit affordance is ambiguous — NF: "is that just the opposite of confirming the result?". Worse, the current "Bestätigt" = acknowledged means "I have seen/acknowledged the anomaly", which collides with the *new* desired meaning "the value is confirmed correct".

### 1.5 NF point 4 — overloaded page (CONFIRMED)
Four stacked cards, two of which (top control + Prüfergebnisse) show the same data; a 5-up meta grid; tabs; an always-expanded amber card. The same flag set appears up to 3×.

### 1.6 Design tokens / components in play (reuse these)
- Card shell: `bg-white dark:bg-gray-800 rounded-xl border border-gray-200 dark:border-gray-700 p-5`.
- Status colors: red (`bg-red-50 text-red-600`), amber (`bg-amber-50 text-amber-600`), green (`bg-green-50 text-green-600`); anomaly card amber-50/amber-200.
- Icons: lucide `AlertCircle, Ban, CheckCheck, CheckCircle2, Circle, Flag, ImageIcon`.
- Primitives available elsewhere: `InfoTooltip` (added in B2, `src/components/...`), `SummaryCard` (QualityPage.tsx:31).
- i18n via `useLanguage().t`. `QualityFlagDialog` for the report-error modal.
- Wire model is **unchanged**: status stays `'open' | 'acknowledged' | 'resolved'`; we only re-map the *labels/semantics* and the *placement*.

## 2. Redesigned information architecture

Collapse four cards → **two cards + one slim header strip**.

```

│ ‹Pseudonym›  [Ausgeschlossen?]      Fortschritt: 3/5 bestätigt    │
│ Alter · Diagnose · Zentrum · Therapie  (one dense meta line)      │
│ [Ausschließen] [Vollständige Fallansicht]                        │
└──────────────────────────────────────────────────────────────────┘
┌─ Card A: Zu prüfende Messwerte  [Tabs: Messwerte | OCT (n)] ──────┐
│ Filter chips: [Alle] [Nur auffällige (n)] [Offen (n)]            │
│ ┌──────────────────────────────────────────────────────────────┐ │
│ │ Param │ Datum │ Wert │ Status        │ Aktion                 │ │
│ │ Visus │ 2024… │ 0.05 │ ● bestätigt   │ [✓ bestätigen][⚑][✎]  │ │
│ │  ⚠ auffällig: Visus < 0.1                 ← inline badge row  │ │
│ │ CRT   │ 2024… │ 612µ │ ⚠ auffällig   │ [✓][⚑ markiert][✎]    │ │
│ │ IVOM  │  —    │  —   │ ⚠ fehlt       │ [⚑ melden]            │ │

└──────────────────────────────────────────────────────────────────┘

The "two redundant approve controls" both disappear. Confirmation now happens **only inline per row** in Card A. Card B keeps the *case-level* "Als geprüft markieren" + an audit/history read-out (who flagged what, when) — a log, not a second editor.

## 3. Inline per-row confirm/flag interaction model

The measurements table becomes the single review surface. Every reviewable item is one row. **Missing-data items get synthetic rows** (parameter, value "—") so they appear in the list, resolving NF point 2 for missing data.

### 3.1 Row anatomy
Columns: **Parameter · Datum · Wert · Status · Aktion**.
- **Status cell** shows the row's current state as a colored pill (see §4): `Offen` / `bestätigt` / `auffällig` / `Behoben`.
- **Inline badge** under the Param/Wert when the row is an anomaly or missing: a small amber/red chip `⚠ auffällig: Visus < 0.1` or `⚠ fehlt: Kein Visus dokumentiert`. This is the inline relocation of the amber card's content (NF point 2).
- **Aktion cell** holds compact icon-buttons, state-dependent:
  - Normal/open row: `✓ Bestätigen` (primary, sets `acknowledged`→shown as "bestätigt"), `⚑ Melden` (opens flag dialog → `open`).
  - Anomaly/flagged row (`open`): `✓ Bestätigen` (accept value as-is), `↑ Behoben` (im Quellsystem korrigiert → `resolved`).
  - Already `bestätigt`/`Behoben`: a single `↺ Zurücksetzen` to revert to open. The active state is visible in the Status cell so the reviewer always sees the current verdict.

### 3.2 State machine (maps onto the existing wire enum — NO schema change)
| UI verdict | Wire `status` | Meaning |
|---|---|---|
| Offen | `open` (flag exists, unresolved) / *no flag* (untouched) | needs review |
| bestätigt | `acknowledged` | reviewer confirms the value is correct as recorded |
| auffällig | `open` **and** anomaly-detected | system-detected anomaly, not yet dispositioned (display-only state, derived) |
| Behoben | `resolved` | reviewer corrected the value upstream in the source EHR ("im Quellsystem korrigiert") |

"auffällig" is a **derived display state** (anomaly present + not yet bestätigt/Behoben). It is not a new wire value; it renders on top of `open`. This keeps the build to label + placement changes, no migration.

### 3.3 Confirm without a pre-existing flag
Today a status `<select>` only exists for items that already have a `QualityFlag`. A normal, non-anomalous measurement cannot be "confirmed". The rework lets the reviewer **confirm any row**: pressing `✓ Bestätigen` on a row with no flag creates a flag with `status: 'acknowledged'` (errorType e.g. `"bestätigt"`/none), so the verdict persists through the same `addQualityFlag`/`updateQualityFlag` path. This is what NF asked for: "confirm results directly in the Messwerte list."

### 3.4 Bulk action
A single footer button `Alle sichtbaren als bestätigt` confirms every currently-visible (filter-respecting) un-dispositioned row in one click — replaces the value the old top approve control provided (fast "approve all") without a duplicate panel.

## 4. Status vocabulary — exact German labels + tooltips

The four user-facing verdicts. **Locked:** keep "Behoben", define it as "im Quellsystem korrigiert".

| Pill label (DE) | Color | i18n key (proposed) | Tooltip (DE) |
|---|---|---|---|
| **Offen** | grey | `statusOpen` (reuse) | „Noch nicht geprüft." |
| **bestätigt** | green | `statusConfirmed` (new; replaces meaning of `statusAcknowledged`) | „Wert wurde als korrekt bestätigt." |
| **auffällig** | amber | `statusAnomalous` (new, derived display) | „Vom System als auffällig erkannt (Schwellenwert über-/unterschritten oder Wert fehlt) – noch nicht beurteilt." |
| **Behoben** | blue | `statusResolved` (reuse key, new tooltip) | „Im Quellsystem korrigiert – der Wert wurde direkt im Quell-EHR berichtigt. Der Demonstrator selbst ändert keine Daten." |

Notes:
- The **action button** that sets `Behoben` is labelled `Im Quellsystem korrigiert` (full, self-explanatory) with the short pill reading `Behoben`. The icon is an "upstream/external edit" glyph (`↑`/`ExternalLink`) — not a checkmark — to visually distinguish it from `Bestätigen`. This directly answers NF point 3 ("not self explanatory, is that just the opposite of confirming?"): Bestätigen = "value is fine as recorded"; Behoben = "value was wrong, I fixed it in the source system".
- EN counterparts: Open / Confirmed / Anomalous / Corrected upstream.
- The current `statusAcknowledged` ("Bestätigt" = acknowledged) is **retired** as a user concept; the wire value `acknowledged` is reused to back the new "bestätigt" verdict (semantics shift from "anomaly seen" → "value confirmed correct"), which is acceptable because no migration of meaning harms existing demo data.

## 5. Inline anomaly / missing-data rendering

- **Anomaly rows** (Visus<thr, CRT>thr, Visus-jump): the offending observation row gets (a) Status pill `auffällig` (amber) and (b) a sub-row badge `⚠ auffällig: ‹reason›` using the existing reason strings (`crtAnomaly`, `visusAnomaly`, `visusJump`). Reuses the amber tokens from the old card.
- **Missing-data rows** (`missingVisus`/`missingCrt`/`missingInjections`): synthetic rows with Wert "—", Status pill `fehlt` (amber, dashed border to read as "absent"), badge `⚠ fehlt: ‹reason›`. Only the `Melden` / `Im Quellsystem korrigiert` actions apply (nothing to "confirm correct").
- The standalone amber card is **removed**. A compact count chip in Card A's filter row (`Nur auffällige (n)`) preserves the at-a-glance "how many anomalies" the card used to give, and filters the table to them.

## 7. Accessibility

- Each action icon-button has an `aria-label` (`Bestätigen ‹Parameter ‹Datum››`, `Im Quellsystem korrigiert ‹…›`, `Fehler melden ‹…›`, `Zurücksetzen ‹…›`).
- Status pills carry `aria-label` with the full verdict word (not only color) — color is never the sole signal; the pill always shows text + an icon.
- Tooltips: `InfoTooltip` primitive (keyboard-focusable, `aria-describedby`), reachable by Tab and shown on focus, not only hover.
- The measurements table is a real `<table>` with `<th scope="col">`; row badges are in a `<td colspan>` sub-row associated with the row above (or `aria-describedby` from the param cell) so screen readers tie the anomaly reason to its measurement.
- Filter chips are a `role="group"` toggle set; bulk button is a normal button with a confirmation-free, reversible action.
- Keyboard: full Tab order Param-row → action buttons → next row; `✓`/`⚑`/`↺` operable via Enter/Space; collapse toggle on Card B is a `<button aria-expanded>`.
