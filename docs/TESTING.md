# Testing

<!-- GSD:GENERATED 2026-04-17 -->

The EMD application uses [Vitest](https://vitest.dev/) (v4.1) as its test runner and
[@testing-library/react](https://testing-library.com/docs/react-testing-library/intro/) (v16)
for React component tests. The test suite contains 47 test files located in the `tests/`
directory at the project root.

## Running tests

Run the full suite once:

```bash
npm test
```

This executes `vitest run`, which runs all files matching `tests/**/*.test.ts` and
`tests/**/*.test.tsx` and exits with a non-zero code on any failure.

Run a single file:

```bash
npx vitest run tests/cohortTrajectory.test.ts
```

Run tests whose names match a pattern:

```bash
npx vitest run --reporter=verbose -t "authMiddleware"
```

Watch mode (re-runs on file changes, useful during development):

```bash
npx vitest
```

## Test environment

The default environment configured in `vitest.config.ts` is `node`, which is appropriate for
server-side and pure-function tests. React component tests require a browser-like DOM. Those
files opt in individually by placing the following docblock on the first line:

```ts
// @vitest-environment jsdom
```

Tests that use this directive also depend on `jsdom` (v29), which is installed as a dev
dependency. Do not omit the directive from `.tsx` test files — the test will run in a `node`
environment otherwise and fail on any DOM access.

#### Recharts mock pattern

Recharts uses `ResizeObserver` internally and does not render meaningfully in jsdom. Every
component test that renders a chart uses the same mock pattern:

```ts
vi.mock('recharts', async (importOriginal) => {
  const real = await importOriginal<typeof import('recharts')>();
  return {
    ...real,
    ResponsiveContainer: ({ children }: { children: any }) => (
      <div data-testid="recharts-responsive-container"><svg>{children}</svg></div>
    ),
    ComposedChart: ({ children }: { children: any }) => (
      <g data-testid="recharts-composed-chart">{children}</g>
    ),
    CartesianGrid: () => null,
    XAxis: () => null,
    YAxis: () => null,
    Tooltip: () => null,
    Legend: () => null,
    ReferenceLine: () => null,
    Area: () => <path data-testid="area-path" d="M0,0 L10,10" />,
    Line: () => <path data-testid="line-path" d="M0,0 L10,10" />,
    Scatter: () => null,
  };
});
```

Spread the real module first (`...real`) so non-chart exports such as type helpers remain
available. Components expose computed values via `data-testid` attributes (for example,
`data-testid="outcomes-panel-ydomain"`) so tests can assert on derived state without parsing
SVG tick labels.

## Writing a new test

1. Create `tests/<subject>.test.ts` for pure logic or `tests/<subject>.test.tsx` for components.
2. Add `// @vitest-environment jsdom` as the first line of any `.tsx` file.
3. Import from `vitest` directly (`import { describe, expect, it, vi } from 'vitest'`).
4. For component tests, copy the Recharts mock block verbatim from an existing `.test.tsx` file
   before importing any component that uses a chart.
5. Place the file in `tests/` — the glob `tests/**/*.test.ts` and `tests/**/*.test.tsx` in
   `vitest.config.ts` picks it up automatically.

Commit message convention for test files: prefix with the relevant plan or requirement ID as a
comment at the top of the file (for example `/** OUTCOME-12: … */`), matching the style used
by existing files.

## Coverage

No automated coverage threshold is configured in `vitest.config.ts`. Coverage is not collected
by default when running `npm test`.

Well-covered areas include server authentication paths, all shared math utilities, server-side
aggregate endpoints, and the i18n translation bundle. Areas intentionally not tested include
pixel-level chart rendering (mocked away), CSS/Tailwind class application, and snapshot-based
UI regression.
